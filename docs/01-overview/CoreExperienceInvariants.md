# 核心体验不变项

以下链路属于当前版本的体验基线，后续重构不得直接破坏：

## 1. 修炼乐观更新（保留）

- 修炼中按秒进行本地乐观增长（灵气、气血、术法使用次数）。
- 本地积累 `pending` 后按策略上报 `cultivation/report`。
- 在关键动作前执行 flush，再进入后续流程（如突破/全量刷新）。

## 2. 炼丹预扣池（保留）

- 多次炼制按“每轮预扣”推进，而不是一次性全扣。
- 单轮结算失败时按当前轮次恢复预扣成本。
- 结束时统一输出“收丹停火：成丹x枚，废丹x枚”。

## 3. 历练 API 回放（保留）

- 战斗过程由服务端模拟结果驱动，客户端按时间轴播放。
- 客户端维护本地历练运行态（是否在历练、当前区域等）以保证切页返回定位。
- 战斗结算失败时执行收敛退出，避免停留在不一致战斗态。

## 4. 业务提示语义码化（保留）

- 成功/失败文案都由 `reason_code` 映射生成。
- `reason_data` 只承载动态数据（数量、资源、ID、结果列表等）。

## 5. 客户端手感优先（保留）

- 客户端体验目标是“服务端返回后，当前页面尽快给玩家可见反馈”。
- 高频动作默认采用“即时反馈优先，全量同步校准”：
  - 先用接口返回值更新当前页面必要状态和日志。
  - 再通过全量同步覆盖跨系统真值。
  - 不在即时阶段猜测复杂 `effect` 带来的跨系统影响。
- 当前页面之外的 UI 刷新应延后、合并或跳过，避免无关页面重建拖慢体感。
- 生产循环、战斗回放、连续操作等节奏型玩法，不得因为非必要 UI rebuild 阻塞下一轮流程。
- 排查手感问题时优先使用 `调试` 频道里的“请求耗时 + 数据同步耗时 + UI rebuild 耗时”三段日志定位。
