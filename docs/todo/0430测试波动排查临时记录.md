# 0430 测试波动排查临时记录

## 背景

在服务端 `app` 目录重构过程中，`unit_test` 全量回归曾出现过少量偶发失败。
目前这些失败没有稳定复现为“单跑必挂”，因此需要先把现象、上下文和当前猜测记录下来，避免后续主线改造时丢失线索。

本记录是临时排查文档，不代表最终结论。

## 涉及到的波动用例

### 1. `test_task_newbie_herb_gather_10_and_20_claim_rewards`

文件：
- `idle_cultivation_server/unit_test/test_task_api.py`

测试场景：
1. 玩家开始采药：`herb_start("point_low_yield")`
2. 连续 10 次：
   - 直接写测试库时间：`set_herb_elapsed_seconds(..., 20.0)`
   - 调一次 `herb_report()`
3. 验证新手任务 `newbie_herb_gather_10`
   - `progress == 10`
   - `completed == true`
   - 领取奖励后应获得 `recipe_health_pill x1`
4. 再连续 10 次同样流程
5. 验证新手任务 `newbie_herb_gather_20`
   - `progress == 20`
   - `completed == true`
   - `claimed == false`
   - 领取奖励后应获得 `spell_alchemy x1`

期望结果：
- 20 次采药累计后，`newbie_herb_gather_20` 必须稳定达到 `20/20`

全量波动时观察到的异常结果：
- 期望：`progress == 20`
- 实际：`progress == 19`

当前代码链路：
- 接口：`idle_cultivation_server/app/game/api/HerbApi.py`
- 任务系统：`idle_cultivation_server/app/game/domain/task/TaskSystem.py`

当前已确认：
- `herb_report()` 成功后会执行：
  - `ctx.task_system.add_progress("daily_herb_count", 1)`
  - `ctx.task_system.add_progress("newbie_herb_gather_10", 1)`
  - `ctx.task_system.add_progress("newbie_herb_gather_20", 1)`
- `TaskSystem.add_progress()` 本身是普通累加并按 target 封顶，代码上没有看到本轮重构直接修改其语义

当前现象判断：
- 全量套跑时偶发少 1 次
- 单独复跑该测试时通过
- 目前更像偶发时序/状态问题，而不是稳定逻辑回归

当前现象判断：

- 说明“每日重置是否按预期触发”这条链路也存在状态/时序型波动

## 为什么现在不能直接判定是本轮重构引起

当前证据更偏向“原有测试链路脆弱点被放大”，原因包括：

1. 本轮重构主要改的是：
   - 代码目录位置
   - import 路径
   - 真值目录与兼容壳边界
2. 没有直接改这些波动用例的核心业务语义：
   - `herb_report()` 的任务累计逻辑
   - `TaskSystem.add_progress()` 的累加逻辑
   - 修炼携带秒数的结算规则
3. 失败形态不是稳定复现：
   - 全量偶发失败
   - 单独复跑通过

这并不能完全排除重构影响，但暂时还没有证据表明“这次改动稳定引入了业务 bug”。

## 当前更合理的几个猜测

### 猜测 1：测试辅助直接写库 + 立刻打接口，存在偶发时序边界

这些用例大量使用：
- `set_herb_elapsed_seconds(...)`
- `set_alchemy_elapsed_seconds(...)`
- `set_cultivation_elapsed_seconds(...)`

这种模式本质上是：
1. 测试直接改数据库状态
2. 立刻发 API 请求
3. 期望服务端用最新状态结算

如果读写上下文、缓存对象或保存时机存在边界，就可能出现“偶发少算一次”。

### 猜测 2：同一 pytest 进程内复用同一测试账号，累计型状态更容易受串扰

虽然之前已经把测试账号隔离到了 `test_<pid>`，避免不同测试进程互踢，但一个全量 pytest 进程里，许多测试仍会复用同一账号命名策略。

对于：
- 任务累计
- 修炼运行态
- 炼丹运行态
- 采药运行态

这类“连续状态型系统”，比普通只读接口更容易暴露串扰。

### 猜测 3：测试服自身启动或稳定性也带来噪音

排查过程中曾遇到过一次：
- `Connection refused`

这说明测试环境本身也存在抖动，不是每一次异常都能直接归因到业务逻辑。

### 猜测 4：这些波动早就存在，只是在重构回归期更容易被看见

因为最近回归跑得更频繁、链路覆盖更密，所以原本低频出现的问题更容易被放大和记录下来。

## 当前临时结论

当前更稳妥的判断是：
- 这些失败值得认真排查
- 但暂时不能直接定性为“本轮服务端重构造成”
- 更像既有测试/状态链路里的偶发时序问题

## 0430 串行 / 并发对照补记

为了验证这些波动是否主要来自“并发测试污染”，额外做了三组对照：

### 1. 固定账号、严格串行、仅跑 4 条可疑用例

测试对象：
- `test_task_daily_reset_triggers_on_write_after_cross_day_while_online`
- `test_task_newbie_herb_gather_10_and_20_claim_rewards`
- `test_task_newbie_craft_health_pill_20_and_claim_recipe`
- `test_cultivation_carry_seconds_accumulates_with_integer_effect_settlement`

执行方式：
- 固定同一测试账号
- 同一服务端实例
- 严格串行
- 连续重复 15 轮

结果：
- `15 / 15` 全部通过

### 2. 固定账号、严格串行、服务端 `unit_test` 全量

执行方式：
- 固定同一测试账号
- 同一服务端实例
- 严格串行
- `unit_test` 全量连续重复 3 轮

结果：
- `3 / 3` 全部通过
- 每轮均为 `68 passed`

### 3. 刻意制造“并发 + 同一账号”场景

执行方式：
- 两个 pytest 进程并发
- 指向同一台服务端
- 强制使用同一个测试账号

结果：
- 可稳定复现异常
- 典型表现：
  - `{'detail': 'KICKED_OUT'}`
  - 任务 / 修炼 / 采药断言链路随之出现“看起来像业务失败”的假象

### 4. 并发但使用不同测试账号

执行方式：
- 两个 pytest 进程并发
- 指向同一台服务端
- 分别使用不同测试账号

结果：
- 两边均通过
- 未复现 `KICKED_OUT`

## 目前更明确的判断

结合以上对照，当前更合理的结论是：

1. 之前记录的这批“波动”里，至少有相当一部分高度怀疑与并发测试污染有关
2. 尤其是“同一账号被两个测试进程并发使用”时，服务端踢下线机制会稳定触发 `KICKED_OUT`
3. 这种情况下，很多后续断言会表现成：
   - 进度少 1
   - `success` 字段缺失
   - 看起来像任务 / 修炼 / 炼丹逻辑偶发失败
4. 但在严格串行、固定账号、同一服务端实例下，目前未能复现这些记录中的 4 条核心波动

因此：
- 这些问题当前更像“并发测试样本被污染后的假红”
- 而不是已经证实存在的串行业务逻辑不稳定

## 当前建议

- 服务端回归继续坚持严格串行
- 如果需要并发跑测试，必须确保不同测试进程使用不同测试账号
- 后续再遇到类似波动，优先先核实：
  - 是否存在并发测试
  - 是否共用了同一测试账号
  - 是否出现 `KICKED_OUT`

## 后续建议排查方向

### 方向 1：对波动链路加临时采样日志

优先顺着：
- `herb_report -> task progress`
- `alchemy_report -> task progress`
- `cultivation_report -> carry_seconds`

加临时日志，记录每一步：
- 结算前状态
- 结算后状态
- 保存前后的 `db_data`
- 任务 progress 变化量

这样最容易抓到“到底哪一次少算”。

### 方向 2：把“直接写库 + 立刻打接口”的测试方式改得更稳

这个方向先不急着做，因为它属于“修测试”。
在还没确认根因前，不应该先用改测试去掩盖问题。

## 当前处理策略

- 先记录，不让线索丢失
- 继续主线重构，不被这类暂时未定性的波动完全打断
- 后续如果波动继续出现，再单独开一个小排查分支或局部临时日志追踪

## 0430 客户端 smoke 失败补记

### 现象

客户端全量测试曾出现过 1 条失败：

- `idle_cultivation_client/tests_gut/integration/TestModuleApiSmoke.gd`
- 用例：`test_module_api_smoke_flow`
- 失败点：任务领奖阶段断言 `harness.last_log().contains("领取成功")`

### 排查结论

这条失败目前已定位为“测试断言过于依赖最后一条日志”的脆弱性，而不是已确认的服务端任务逻辑错误。

任务领奖调用链是：
1. `TaskModule._on_claim_pressed("newbie_open_starter_pack_1")`
2. 成功后先发出领奖成功日志
3. 随后继续执行：
   - `_refresh_task_list()`
   - `game_ui.refresh_all_player_data()`
   - `game_ui.update_ui()`

由于成功日志后面还会发生额外刷新，`last_log()` 偶发被后续日志覆盖，导致：
- “领奖实际上成功了”
- 但“最后一条日志不再是领奖成功”
- 进而让 smoke 误报失败

### 已做处理

已将这类断言从：
- “最后一条日志必须包含 `领取成功`”

调整为：
- “本轮日志序列中出现过 `领取成功` 即可”

同步修改了：
- `idle_cultivation_client/tests_gut/integration/TestModuleApiSmoke.gd`
- `idle_cultivation_client/tests_gut/unit/ui/TestTaskModuleApi.gd`

### 复测结果

串行复测结果：
- `TestTaskModuleApi` 通过
- `TestModuleApiSmoke` 通过
- 客户端全量 `182/182 passed`

### 当前判断

这次客户端 smoke 失败不再视为服务端主线业务波动。
它更像是：
- 领奖成功后的异步刷新引发日志顺序变化
- 测试断言方式过于依赖“最后一条日志”的偶发脆弱点
