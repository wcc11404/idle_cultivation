# 0429 模块消息提醒系统设计（一期：任务 + 邮箱）

## 1. 目标与命名

### 1.1 系统名称
- 中文名：`模块消息提醒系统`
- 英文内部名（建议）：`Module Notification System`

### 1.2 一期目标
- 基于现有 30 秒轮询结果，在客户端提供统一“红点提醒”能力。
- 支持“二级页面提醒 -> 一级入口按钮提醒 -> Tab 按钮提醒”的透传链路。
- 一期先接入：
  1. `任务（仙务司）`
  2. `邮箱（设置页邮箱按钮）`

### 1.3 二期扩展目标（预留）
- 后续模块（经脉、活动、商城、福利等）无需重复造轮子，按统一接口接入。
- 保持提醒逻辑与 UI 渲染解耦，避免各模块直接互相依赖。

---

## 2. 术语与标准定义

## 2.1 红点（Badge）
- 视觉形态：右上角小红点（一期默认不显示数字）。
- 一期不做“红叹号/数字角标”，保留扩展位。

## 2.2 节点层级
- 一级入口：
  - 主 Tab（例如：`地区`、`设置`）
- 二级入口：
  - 一级页内功能按钮（例如：`仙务司`、`邮箱`）

## 2.3 透传定义
- “透传”指：只要二级入口存在未处理提醒，一级入口同步亮红点。
- 同一一级入口下多二级来源取 OR 关系（任一为 true 即显示）。

## 2.4 状态真值来源
- 统一以服务端列表接口返回为准：
  - 任务：`task_list`
  - 邮箱：`mail_list`
- 本地不推导业务真值，仅做状态缓存与渲染。

---

## 3. 一期业务规则

## 3.1 任务红点规则
- 条件：存在至少一个任务满足 `completed = true && claimed = false`
- 影响：
  - `地区页 -> 仙务司按钮` 显示红点
  - `主 Tab -> 地区` 显示红点
- 即时性要求：
  - 当用户领取任务奖励后，必须立即重新拉取 `task_list` 并重算可领取数。
  - 若领取后剩余可领取任务数为 `0`，红点应当立即熄灭（不等待 30 秒轮询）。

## 3.2 邮箱红点规则
- 条件：`mail_list.unread_count > 0`
- 影响：
  - `设置页 -> 邮箱按钮` 显示红点
  - `主 Tab -> 设置` 显示红点
- 字段约定：
  - 继续使用服务端 `mail_list` 返回的 `unread_count` 作为未读真值，不在客户端遍历邮件列表重复计算。
  - 该字段用于红点判断与轻量轮询，避免列表较大时客户端重复扫描。

## 3.3 清除时机
- 任务：领奖成功后刷新任务列表，若无可领任务则熄灭。
- 邮箱：查看详情标记已读或领取/删除后刷新，若无未读则熄灭。
- 轮询失败不清空状态，保持上一次成功状态，避免闪烁。

## 4. 客户端架构方案

## 4.1 新增聚合器（建议）
- 文件建议：`scripts/ui/common/NotificationBadgeState.gd`
- 职责：
  - 持有各提醒源布尔状态（任务、邮箱等）。
  - 按层级输出聚合状态（一级/二级）。
  - 提供信号通知 UI 更新（避免 UI 主动轮询状态对象）。

建议数据结构：
```gdscript
{
  "task_claimable_count": int,
  "mail_unread_count": int,
  "task_claimable": bool,  # task_claimable_count > 0
  "mail_unread": bool,     # mail_unread_count > 0
  "region_tab_badge": bool,   # task_claimable
  "settings_tab_badge": bool  # mail_unread
}
```

## 4.2 状态更新入口
- 统一由 `GameUI` 当前 30 秒轮询链路更新：
  1. `task_list`
  2. `mail_list`
- 并在以下时机做“局部即时刷新”：
  - 任务领取成功后
  - 邮箱详情/领取/删除/一键删除后
- 轮询精简建议：
  - 邮箱保留 `mail_list.unread_count` 摘要字段（有实际用途）。
  - 任务侧建议补充 `refresh_indicator_only()` 风格入口，仅提取可领取计数用于红点更新。

## 4.3 与模块边界
- `TaskModule`、`MailModule` 只负责上报“本模块最新摘要状态”到聚合器，不直接改 Tab 红点。
- `GameUI` 负责把聚合器状态映射到具体节点显示/隐藏。

## 5. UI 设计规范

## 5.1 红点样式（一期）
- 颜色：高可见红（建议 `#E74C3C` 附近）
- 大小：
  - Tab 级：10~12 px
  - 二级按钮级：8~10 px
- 描边：1px 浅边（可选）提升浅色背景识别度
- 层级：红点应在按钮内容上层显示，不遮挡主文案

## 5.2 位置规范
- Tab 按钮红点：按钮右上角内缩 4~6 px
- 二级入口按钮红点：按钮右上角内缩 4~6 px
- 同类入口位置统一，避免视觉跳动

## 5.3 交互规范
- 红点仅作为“状态提示”，不拦截点击。
- 点击按钮进入页面后不强制立即清除，仍以“服务端状态刷新结果”为准。

## 7. 错误与边界处理

- 轮询失败：不清空已有红点状态，等待下一轮。
- 接口超时：保持上次状态，避免错误闪烁。
- 用户登出：
  - 清空本地红点状态缓存
  - 下次登录从“全部熄灭 + 首次轮询更新”开始
- 账号切换：
  - 同登出处理，防止红点串号

## 8. 测试方案

## 8.1 逻辑测试
- 任务列表含可领奖 -> `仙务司` + `地区Tab` 红点亮
- 邮箱未读 > 0 -> `邮箱按钮` + `设置Tab` 红点亮
- 两者都满足 -> 四处同时亮
- 两者都不满足 -> 全灭

## 8.2 刷新测试
- 轮询更新前后状态变化正确
- 轮询失败不误清空
- 任务领取后红点熄灭（需刷新后）
- 邮箱读信后红点熄灭（需刷新后）

## 8.3 UI 测试
- 红点位置不遮挡文字
- 各分辨率/手机端下位置稳定
- 不影响按钮点击与滚动

## 9. 后续扩展约定

- 新模块接入只需实现：
  1. 产生一个布尔提醒源（如 `jingmai_reward_available`）
  2. 声明映射到哪个二级入口和一级入口
- 不允许新模块直接操作 Tab 红点节点，必须经聚合器统一输出。

---

## 10. 一期结论

- 一期采用“任务 + 邮箱”双源接入，完成从二级入口到主 Tab 的红点透传闭环。
- 方案重点是“统一聚合 + 统一渲染 + 统一刷新”，避免后续模块增加时逻辑分散。
