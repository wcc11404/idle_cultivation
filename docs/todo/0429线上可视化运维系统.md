# 0429 线上可视化运维系统设计（一期可开发版）

## 1. 文档目的

这份文档用于明确“一期线上可视化运维系统”该怎么落地，重点回答四个问题：

1. 这是不是前后端分离方案
2. 前后端分别用什么语言和框架
3. 代码放在哪里、接口如何分层
4. 一期先做哪些，哪些暂时不做

目标不是一次把所有运维能力做满，而是先做出一套：

- 能登录
- 能看玩家
- 能发放
- 能管控
- 能审计
- 能安全上线

的一期系统。

---

## 2. 结论先说

## 2.1 是否前后端分离

是，但属于“同仓库、同机部署”的前后端分层方案，不建议一期拆成两套完全独立的仓库。

更准确地说：

- 前端：单独一个运维网页
- 后端：单独一层 `/ops/*` 运维 API
- 部署：仍和现有游戏服务端放在同一台机器上
- 代码：建议都放在 `idle_cultivation_server` 仓库里维护

这意味着：

- 前端和后端职责分离
- 运维前端不直接碰数据库
- 运维前端也不直接调用现有 `/api/game/*`
- 所有运维操作都统一经过 `/ops/*`

## 2.2 推荐技术选型

结合当前项目现状，一期推荐技术栈如下：

- 运维后端：`Python + FastAPI`
- 运维前端：`TypeScript + React + Vite`
- 静态资源托管：`Nginx` 或 `FastAPI StaticFiles`
- 数据库存储：沿用当前服务端数据库（PostgreSQL）

推荐原因：

- 现有游戏服务端已经是 `FastAPI`
- 现有早期管理接口实现可以作为迁移参考
- 运维网页需要表格、筛选、批量操作、确认流，`React + Vite` 更适合长期维护
- 一期不建议用 Godot 做运维工具页
- 一期也不建议直接用 Streamlit 作为正式方案，虽然开发快，但后续权限、审计、复杂交互会比较吃力

---

## 3. 一期目标与边界

## 3.1 系统目标

为线上环境提供一套“网页登录可视化运维系统”，让有权限的管理员可以通过浏览器完成以下工作：

- 查看服务状态
- 查看玩家列表与玩家详情
- 封禁 / 解封玩家
- 强制玩家下线
- 单发 / 群发物品
- 单发 / 群发邮件
- 查看关键操作审计日志
- 管理禁止登录状态

## 3.2 一期必须做到

- 可登录
- 可鉴权
- 可做权限校验
- 可做高危操作确认
- 可记录审计日志
- 可区分成功与失败原因
- 支持全服群发且入口足够直接
- 只允许发物品、发邮件，不允许直接改资源数值
- 只做永久封禁，但必须支持解除封禁
- 禁止登录时保留白名单测试账号

## 3.3 一期暂不做

- BI 报表
- 复杂图表大盘
- 自动活动编排
- 多角色复杂审批流
- 短信 / OTP 二次验证
- 完整在线行为分析
- 多环境统一总控台
- 真正执行服务器重启
- 为运维系统单独推动 Nginx 升级改造

## 4. 当前项目基础与现实约束

## 4.1 当前已有基础

当前 `idle_cultivation_server` 已经具备以下条件：

- 服务端技术栈是 `FastAPI`
- 已存在早期管理接口实现，当前真值在 `app/admin_support/api/AdminApi.py`
- 已具备部分管理能力，例如：
  - 管理员登录
  - 玩家列表
  - 玩家详情
  - 封号
  - 发邮件

因此，一期不应推倒重做，而应在现有服务端基础上把这些早期能力迁移并收敛到正式的 `ops_*` 体系里。

## 4.2 当前明显不足

现有 `Admin.py` 更像“早期管理工具接口”，还不是正式运维平台，主要问题包括：

- 管理员账号目前是硬编码
- 权限模型不完整
- 审计体系缺失
- 高危操作缺少二次确认
- 前端操作流尚不存在
- 批量能力不完整
- 系统级维护能力尚未收口

所以一期的核心，不是简单“再写个网页”，而是把这套能力治理起来。

## 5. 总体架构

## 5.1 三层结构

建议采用三层结构：

### A. 运维平台 API 层

位置：

- `app/ops_api/*`

职责：

- 对前端提供正式运维接口
- 处理鉴权、参数校验、权限校验、错误格式
- 不直接堆积复杂业务逻辑

典型能力：

- 运维登录
- 查玩家
- 发邮件
- 发物品
- 封禁 / 解封
- 强制下线
- 开关登录闸门
- 踢出所有玩家
- 维护白名单测试账号

### B. 运维内部服务层

位置建议：

- `app/ops_services/*`

职责：

- 承接运维业务编排
- 复用游戏业务模块能力
- 处理 preview / confirm、高危确认、批量动作和状态聚合
- 向审计层写入审计记录

### C. 运维支撑层

位置建议：

- `app/ops_auth/*`
- `app/ops_audit/*`
- `app/ops_models/*`

职责：

- 运维账号与 token 管理
- 审计日志落库
- 运维系统状态与白名单表管理

### D. 运维平台前端层

位置建议：

- `ops_web/*`

职责：

- 登录页
- 玩家管理页
- 发放页
- 审计页
- 系统维护页

这层只关心页面交互，不直接连接数据库，也不直接调用 `/api/game/*` 或内部模块。

## 5.2 调用关系

标准链路如下：

`Browser -> ops_web -> /ops/api/* -> ops_services/* -> 业务模块/数据库`

重点约束：

- 运维前端只能调 `/ops/api/*`
- 对外只保留一套正式运维入口
- 不再保留 `/api/admin/*` 作为正式方案
- 不允许运维前端直接连数据库
- 运维接口不能假设“只有前端页面会访问”，必须始终做鉴权与权限校验

---

## 6. 为什么这样分层

这样设计的核心收益有四个：

### 6.1 安全边界更清晰

如果同时保留 `/api/admin/*` 和 `/ops/*` 两套公开入口，权限、审计、确认逻辑容易重复，后面会越来越难控。

统一收口到 `/ops/api/*` 后：

- 权限统一
- 审计统一
- 错误格式统一
- 确认流程统一

### 6.2 复用已有能力，避免重复造轮子

现有 `Admin.py` 已经做了一部分能力，一期最好迁移并复用已有实现思路，而不是再并行维护两套正式后台接口。

### 6.3 便于后续扩展

以后如果要加：

- 多角色权限
- 审批流
- 风险拦截
- 批量模板
- 自动化脚本

都可以继续长在 `/ops/*` 这一层，而不是把 `Admin.py` 越堆越重。
都可以继续长在 `ops_*` 体系里，而不是继续维护一套历史 `Admin.py` 路由。

### 6.4 便于长期运维

运维系统本质上是高危系统。长期看，治理能力比“页面能不能点”更重要。

## 7. 一期技术选型说明

## 7.1 运维后端

推荐：

- 语言：`Python`
- 框架：`FastAPI`

原因：

- 和现有服务端同技术栈
- 可直接复用现有鉴权、日志、配置、数据库初始化流程
- 可与当前项目部署方式保持一致

## 7.2 运维前端

推荐：

- 语言：`TypeScript`
- 框架：`React`
- 工程工具：`Vite`
- UI 组件：一期可先轻量自写，或后续补组件库

原因：

- 表格、表单、筛选、分页、批量选择都比较成熟
- 开发体验和后续维护性较好
- 页面组织比服务端模板渲染更清晰
- 比 Streamlit 更适合做正式运维后台

## 7.3 为什么不建议一期用服务端模板

服务端模板渲染不是不能做，但不太适合这个项目的一期方向，主要原因：

- 后台会有较多交互状态
- 批量操作、二次确认、分页筛选会更繁琐
- 后面演进为正式后台时，模板页通常还得重构

## 7.4 为什么不建议运维前端直接放到客户端仓库

运维系统本质上服务于服务器运维，不属于游戏客户端的一部分。

所以建议：

- 设计稿可以写在客户端 `docs/todo/`
- 但真正代码落地放到 `idle_cultivation_server`

这样更符合职责边界。

## 9. 账号、鉴权与权限模型

## 9.1 运维账号

一期先支持一类账号：

- `super_admin`

它具备所有权限，但实现上仍建议按权限点建模，不要直接写死成“只有一个管理员所以不做权限”。

## 9.2 建议预留的权限点

- `player.view`
- `player.ban`
- `player.kick`
- `grant.item`
- `grant.mail`
- `system.view`
- `system.control`
- `audit.view`

## 9.3 鉴权建议

- 登录成功后返回运维专用 token
- 运维 token 和玩家 token 必须完全分离
- token 应有过期时间
- 服务端应支持主动失效 / 撤销
- `/ops/api/*` 是对外唯一运维入口

## 9.4 最低安全要求

- 密码必须哈希存储
- 登录失败做限流
- 高危操作需二次确认
- 所有写操作必须记审计
- 生产环境必须启用 HTTPS
- 审计日志保留时长：永久

- 但必须做明显的风险提示与二次确认

- 正文长度限制

支持：

- 单人封禁
- 单人解封
- 批量封禁

- 批量强制下线

一期建议：

- 只支持永久封禁
- 必须支持解除封禁
- 不做定时封禁

## 10.7 禁止登录与维护入口

一期先做“入口与状态管理”，不执行真实重启动作。

支持：

- 开启禁止登录
- 关闭禁止登录
- 配置或查看白名单测试账号
- 预留“踢出所有在线玩家”入口
- 预留“重启服务器”入口

一期边界：

- “踢出所有在线玩家”可先不接真实动作，视开发节奏决定
- “重启服务器”只保留页面入口和后端占位，不调用任何实际功能
- 白名单测试账号在禁止登录期间仍可进入游戏

约束：

- 仅 `super_admin`
- 必须二次确认
- 必须有互斥锁
- 必须写审计

## 10.7.1 登录闸门 + 白名单实现方案

一期明确采用“服务端登录闸门”方案，不在客户端或运维前端做安全拦截。

核心原则：

- 是否允许登录，由游戏服务端登录接口最终判定
- 运维前端只负责配置开关和白名单，不负责真正放行
- 白名单账号在“禁止登录”开启时仍可登录
- 白名单判断按 `account_id` 进行，不按昵称判断

建议登录链路：

1. 客户端发起登录请求
2. 服务端校验账号密码
3. 服务端校验账号是否被永久封禁
4. 服务端读取 `ops_system_state.login_gate_enabled`
5. 若 `login_gate_enabled = false`，正常放行登录
6. 若 `login_gate_enabled = true`，继续检查 `ops_login_whitelist`
7. 若当前账号在白名单中，允许继续登录
8. 若当前账号不在白名单中，拒绝登录并返回明确业务错误码

建议错误码：

- `LOGIN_DISABLED`
- `LOGIN_DISABLED_NOT_IN_WHITELIST`

建议前端提示文案：

- `服务器维护中，当前账号未开放登录`

运维后台需要提供的能力：

- 开启禁止登录
- 关闭禁止登录
- 查看当前白名单
- 添加白名单账号
- 移除白名单账号
- 查看最近一次闸门状态变更人与时间

审计要求：

- 开启禁止登录必须记录审计
- 关闭禁止登录必须记录审计
- 白名单增删必须记录审计
- 白名单测试账号登录本身不必写审计日志，但白名单配置变更必须写审计

## 11. `ops_api` 与 `ops_services` 的职责划分

## 11.1 `ops_api` 的定位

`ops_api` 是正式运维接口层。

它负责：

- 接收前端请求
- 做鉴权与权限校验
- 做参数校验
- 做统一返回格式
- 调用内部服务层

## 11.2 `ops_services` 的定位

`ops_services` 是内部运维服务层。

它负责：

- 编排具体运维动作
- 复用游戏业务模块能力
- 处理批量逻辑与高危确认流
- 调用审计与状态存储能力

## 11.3 一期原则

- 对外只保留 `/ops/api/*`
- 不再新增 `/api/admin/*` 能力
- 现有 `Admin.py` 里的可用逻辑逐步迁移到 `ops_services/*`
- 后续所有运维能力统一收敛到 `ops_*` 命名空间

一句话说：

`ops_api` 管“接口边界”，`ops_services` 管“具体动作怎么执行”。

## 12. 推荐接口清单

## 12.1 运维后端 API

建议新增：

- `POST /ops/auth/login`
- `POST /ops/auth/logout`
- `GET /ops/auth/me`
- `GET /ops/system/health`
- `GET /ops/system/summary`
- `GET /ops/players`
- `GET /ops/players/{account_id}`
- `POST /ops/players/ban`
- `POST /ops/players/unban`
- `POST /ops/players/kick`
- `POST /ops/grant/items/preview`
- `POST /ops/grant/items/confirm`
- `POST /ops/grant/mails/preview`
- `POST /ops/grant/mails/confirm`
- `GET /ops/audit/list`
- `POST /ops/system/login-gate`
- `GET /ops/system/whitelist`
- `POST /ops/system/whitelist`
- `POST /ops/system/kick-all`（可先占位）
- `POST /ops/system/restart`（一期只占位，不执行实际动作）

## 12.2 为什么推荐 preview / confirm 两段式

因为发物品、发邮件、批量封禁、踢全服、重启都属于高风险动作。

两段式的好处是：

- 先校验参数
- 先告诉操作者影响范围
- 降低误操作概率
- 更方便审计记录

如果一期不想全部两段式，至少这些操作应两段式：

- 批量发物品
- 批量发邮件
- 全服发物品
- 全服发邮件
- 禁止登录开关
- 踢出所有玩家

## 13. 建议补齐的 `ops_services` 能力

一期建议在 `ops_services/*` 内补齐这些能力，供 `/ops/api/*` 调用：

### 13.1 玩家查询

## 14.2 全局系统状态表 `ops_system_state`

这张表不是一期绝对必需，但建议预留，用于支持运维 token 的主动失效、登出和后续安全治理。

建议字段：

- `id`
- `user_id`
- `token_jti`
- `expires_at`
- `revoked_at`

- `confirm_token`
- `request_payload`

## 14.7 审计要求

- 审计日志永久保留，因此需要从一开始就考虑分页、索引和筛选字段

## 19. 风险点与建议

## 19.1 最大风险不是页面，而是高危写操作治理

这个系统最容易出事的地方不是“列表显示错了”，而是：

- 发错全服奖励
- 封错玩家
- 误踢全服
- 误重启

所以优先级应始终是：

安全 > 审计 > 权限 > 确认流 > 页面体验

## 19.2 一期不做真实重启动作

建议只做：

- 重启入口占位
- 后续实现预留

一期不做：

- 游戏服务进程重启
- 整台机器重启

原因是当前目标是先把运维入口、权限、审计、发放和玩家管控跑通，不必提前引入更高风险动作。

## 19.3 Nginx 不是一期前置条件

Nginx 统一入口是一个合理方向，但它属于整体部署体系优化，不只是运维系统自身升级。

所以一期可以先：

- 把运维后端和前端结构搭好
- 先用后端托管静态资源或简单部署方式跑通
- 后续再统一收口到 Nginx

## 19.4 不建议一期承诺强一致在线人数

如果当前还没有稳定在线会话中心，一期首页可以展示：

- 近实时活跃人数
- 最近在线玩家数

不要先把文案写成强一致“当前在线”。

## 19.5 一期不要直接开放太多管理员

建议先只开：

- `super_admin`

等系统稳定后，再扩展普通运维角色。

## 20. 已拍板项

以下事项已明确，不再作为开放问题：

1. 群发允许全服所有玩家，而且要做成清晰直接的正式入口
2. 封禁只做永久封禁，但必须支持解除封禁
3. 不允许运维直接改资源数值，只允许发物品、发邮件
4. 禁止登录时保留白名单测试账号
5. `login_gate_enabled` 采用方案 A，存入独立的全局系统状态表 `ops_system_state`
6. 白名单账号采用独立表 `ops_login_whitelist` 管理，并按 `account_id` 判断
7. 审计日志永久保留
8. 重启功能一期只保留入口占位，不执行任何实际动作
9. Nginx 统一入口不是一期前置条件，作为后续整体部署优化项
10. 现有 `Admin.py` 不再作为正式长期方案保留，对外只保留 `ops_*` 一套体系

## 22. 最终建议

一期推荐方案可以概括为一句话：

在 `idle_cultivation_server` 内新增一套 `Python + FastAPI` 的 `ops_*` 运维体系，并配一个 `TypeScript + React + Vite` 的运维网页前端；前端只调 `/ops/api/*`，后端通过 `ops_api/* + ops_services/* + ops_auth/* + ops_audit/* + ops_models/*` 统一承接权限、确认、审计、发放、封禁/解封和禁止登录白名单能力。现有 `Admin.py` 作为早期实现参考逐步迁移，不再作为长期正式入口保留；重启功能先只保留入口占位，Nginx 统一入口留作后续整体部署优化。

如果后面按这个稿子推进，开发上会比较顺：

- 方向清楚
- 分层清楚
- 风险边界清楚
- 一期优先级也清楚

这份文档的目标就是把这些边界先钉住。
