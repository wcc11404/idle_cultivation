# 0430 服务端 `app` 代码重构方案

## 1. 文档目标

这份文档不是讨论“理想架构图有多漂亮”，而是给当前 `idle_cultivation_server/app` 做一套可落地、可分阶段执行、风险可控的重构方案。

目标有三个：

1. 把现有 `app` 目录职责重新讲清楚
2. 给出一套长期更清晰的目录结构
3. 给出从现状迁移到目标结构的阶段性方案

## 2. 先说结论

我建议这次服务端先收敛成两大主域加一个共享层：

- `app/core`：共享基础设施
- `app/game`：游戏业务域

也就是：

```text
app/
  core/
  game/
```

这是我最推荐的这轮重构目标结构。

原因很简单：

- `core` 只放真正共享的底层能力
- `game` 一眼就是玩家游戏逻辑
- 后面如果真要接运维系统，也能自然继续扩别的域

比如未来如果有：

- `analytics`
- `gm_tools`
- `matchmaking`
- `ops`

都可以继续按“领域目录”长出来，而不是继续堆在 `modules`、`api`、`core` 这种历史型目录里。

## 3. 现状梳理

当前服务端 `app` 的主要结构是：

```text
app/
  api/
  core/
  db/
  modules/
  resources/
  schemas/
  main.py
```

### 3.1 现有目录职责

从代码看，当前职责大致是：

- `app/api`
  - 对外 HTTP 路由入口
  - 目前有 `GameBase.py`、`Admin.py`、`Test.py`
- `app/modules`
  - 游戏功能模块
  - 每个模块通常同时放：
    - `XxxApi.py`
    - `XxxSystem.py`
    - `XxxData.py`
    - `json` 配置资源
- `app/core`
  - 共用基础能力
  - 但已经混入不少“半业务编排”逻辑
- `app/db`
  - Tortoise 模型与数据库初始化
- `app/schemas`
  - Pydantic 请求/响应结构

这个结构不是错，但已经明显进入“项目长大后边界开始模糊”的阶段。

## 4. 当前主要问题

## 4.1 `app/core` 过宽，已经不只是基础设施

当前 `core` 里有：

- `ServerConfig.py`
- `Logger.py`
- `Security.py`
- `WriteLock.py`
- `SensitiveWordFilter.py`
- `Dependencies.py`
- `DailyReset.py`
- `InitPlayerInfo.py`
- `AntiCheatSystem.py`
- `Validator.py`
- `TestAccountSeeder.py`

这说明 `core` 已经混了至少四类东西：

1. 基础设施
2. 认证安全
3. 游戏运行时编排
4. 测试辅助

典型例子：

- `ServerConfig.py`、`Logger.py` 很像基础设施
- `Security.py` 属于安全能力
- `Dependencies.py` 其实已经是“游戏应用层装配”
- `DailyReset.py`、`InitPlayerInfo.py` 明显带游戏语义
- `TestAccountSeeder.py` 是测试启动辅助

这会带来一个结果：

`core` 这个名字本来应该让人觉得“很稳、很底层、可复用”，但现在它已经像一个混合抽屉。

## 4.2 `modules` 里混了 API、领域逻辑、配置数据

比如 `task` 模块里同时有：

- `TaskApi.py`
- `TaskSystem.py`
- `TaskData.py`
- `tasks.json`

这在项目早期很方便，但中后期会出现几个问题：

- 领域逻辑和接口层耦合感变强
- 模块之间风格不够统一
- API 注册入口分散
- 后续要抽 service/application 层时比较别扭

而从当前重构进度看，这一问题已经开始被真正消解：

- `app/game/domain` 已经承接了主线真值
- `app/modules/*` 当前越来越接近“兼容导出目录”
- 测试代码也已经开始直接依赖 `app/game/domain`

## 4.3 `api` 目录既是游戏接口入口，又混着测试和历史管理接口

当前 `app/api/__init__.py` 会注册：

- `auth`
- `game`
- `admin`
- `test`

这说明 `api` 现在既不是纯游戏域，也不是纯系统级入口层，而是“所有顶层路由都先塞这里”。

这会让后续继续加运维、管理、内部工具时越来越乱。

## 4.4 `schemas` 是按响应类型聚合，不是按领域聚合

当前是：

- `Auth.py`
- `Game.py`
- `Test.py`

这在体量小的时候没问题，但随着接口增长，会变成：

- `Game.py` 无限膨胀
- 单文件塞太多请求/响应定义
- 很难按领域追踪接口契约

这一块当前也已经进入“旧目录临时保留、真值迁走”的状态：

- `app/game/schemas/*Schema.py` 已经成为主线真值
- `app/schemas/*` 当前主要是兼容导出层

## 4.5 游戏“应用层”目前不够明确

从 `core/Dependencies.py` 可以看出，当前有一个很关键的 `GameContext`：

- 读玩家档案
- 反序列化多个 system
- 组装运行时对象
- 提供 save

它其实已经非常像“游戏应用层装配器”了。

但它现在挂在 `core/` 下，会让目录语义失真。

## 4.6 测试与运维辅助能力没有自己的归属

比如：

- `Test.py`
- `TestAccountSeeder.py`

这些东西不是游戏主线逻辑，但也不是基础设施。

如果后面继续增长，就会进一步加重 `core` 和 `api` 的职责混乱。

## 5. 重构目标

## 5.1 目标不是“全项目大改名”

这次重构的目标不是追求形式整齐，而是要解决三个实际问题：

1. 新人看目录时，能快速判断代码该放哪
2. 游戏 API、游戏业务、共享基础设施三者边界更清晰
3. 游戏 API、游戏业务、共享基础设施三者边界更清晰

更准确地说，这次文档聚焦的是：

- 先把现有游戏服务端理顺
- 先不把运维系统实现细节纳入这份重构主线

## 5.2 目标结构

推荐最终收敛为：

```text
app/
  core/
    config/
    logging/
    security/
    db/
    locks/
    resources/

  game/
    api/
    application/
    domain/
    content/
    schemas/
    bootstrap/

  main.py
```

这不是要求一次到位，但应该作为明确终态。

## 5.3 命名规则建议

为了避免后续出现大量同名文件，比如：

- `api/cultivation.py`
- `schemas/cultivation.py`
- `domain/cultivation/`
- `content/cultivation/`

这轮重构建议统一采用“领域名 + 层后缀”的命名规则。

推荐规则：

- API 层文件：`XxxApi.py`
- Schema 层文件：`XxxSchema.py`
- Domain 层：使用领域目录名，目录内再按职责命名文件
- Content 层：使用领域目录名，目录内放资源文件

例如“修炼”模块推荐写成：

- `game/api/CultivationApi.py`
- `game/schemas/CultivationSchema.py`
- `game/domain/cultivation/`
- `game/content/cultivation/`

这样做的好处：

- 同一领域词仍然保持一致
- 看到文件名就知道它属于哪一层
- 避免多个平级文件都叫 `cultivation.py`
- 搜索领域名时仍然容易聚合

## 6. 各层职责定义

## 6.1 `app/core`

只放共享基础设施，不放具体游戏语义。

适合放这里的内容：

- 配置
- 日志
- JWT / 通用安全工具
- DB 初始化
- 分布式锁 / 写锁
- 通用文本资源加载

不适合继续放这里的内容：

- 每日重置规则
- 初始化玩家数据
- 游戏上下文装配
- 测试账号播种

换句话说，`core` 应该尽量“去游戏语义化”。

这一步当前已经开始落地，至少最典型的基础设施文件已经先下沉到了：

- `core/config`
- `core/logging`
- `core/security`
- `core/locks`

这样第二轮不再只是概念设计，而是已经开始把“共享基础设施”和“游戏应用编排”真正拉开。

当前已经明确不适合继续留在 `core` 的，还有：

- `AntiCheatSystem.py`

它虽然名字里带 `System`，但实际职责是围绕账号运行态、玩家存档、踢下线策略做游戏应用编排，而不是共享基础设施。因此这轮已经先迁到 `game/application`，旧路径保留兼容层。

## 6.2 `app/game`

这是最大的业务域，承接当前绝大多数内容。

建议拆成：

### `game/api`

只放游戏对外 HTTP 路由。

比如：

- `AuthApi.py`
- `CultivationApi.py`
- `SpellApi.py`
- `InventoryApi.py`
- `AlchemyApi.py`
- `LianliApi.py`
- `HerbApi.py`
- `TaskApi.py`
- `MailApi.py`
- `BaseApi.py`

### `game/application`

放游戏应用层编排。

这是当前代码最缺但最需要补清楚的一层。

适合迁到这里的内容：

- 当前 `core/Dependencies.py`
- 当前 `core/DailyReset.py`
- 当前 `core/InitPlayerInfo.py`
- 统一的游戏上下文装配与保存
- 登录/登出时的运行态重置

也就是说，`GameContext` 最终不应该继续在 `core`，而应该在：

- `app/game/application/GameContext.py`

### `game/domain`

放纯游戏规则与状态模型。

可按模块拆子目录：

```text
game/domain/
  account/
  player/
  cultivation/
  spell/
  inventory/
  alchemy/
  lianli/
  herb/
  task/
  mail/
```

每个模块里再放：

- state / system
- rule / service
- static data loader

例如当前：

- `TaskSystem.py`
- `TaskData.py`

未来可迁成：

```text
game/domain/task/System.py
game/domain/task/Data.py
```

这里不建议再放裸的 `task.py`、`cultivation.py` 这类文件，而是按职责命名：

- `System.py`
- `Rules.py`
- `Service.py`
- `Models.py`
- `DataLoader.py`

### `game/content`

放静态配置资源：

- `tasks.json`
- `items.json`
- `recipes.json`
- `spells.json`
- `realms.json`
- `areas.json`
- `enemies.json`
- `herb_points.json`

这样可以把“数据文件”和“领域代码”分离开。

### `game/schemas`

按游戏领域拆请求/响应结构，而不是只放一个 `Game.py`。

例如：

```text
game/schemas/
  CommonSchema.py
  AuthSchema.py
  CultivationSchema.py
  InventorySchema.py
  SpellSchema.py
  AlchemySchema.py
  LianliSchema.py
  HerbSchema.py
  TaskSchema.py
  MailSchema.py
```

### `game/bootstrap`

放游戏启动期或运行期初始化辅助。

适合放：

- 测试账号初始化
- 敏感词初始化
- 游戏内容预热

不过如果某些能力是全局共享的，也可以继续留在 `core`。

## 6.3 `app/test_support`

测试辅助逻辑建议逐步独立出来，不继续混在 `core` 或顶层 `api`。

适合放这里的内容：

- 测试专用 API
- 测试账号播种
- 测试预置数据装配
- 只服务于自动化测试的辅助逻辑

## 8. 为什么我推荐 `app/core + app/game`

## 8.1 比当前结构更清楚

当前结构的问题不是不能工作，而是目录语义依赖“项目老人经验”。

新结构能做到：

- 共享层一眼看懂
- 游戏层一眼看懂

## 8.2 比“继续维持旧结构”更长期

如果只是继续沿用当前：

- `api`
- `modules`
- `core`

短期当然能跑，但长期会继续出现：

- `core` 膨胀
- `api` 混域
- `modules` 同时装 API、规则、内容资源

## 8.3 比“全量 rename 一次到位”更稳

我不建议直接对现有所有文件名做激进改名，而是建议：

- 先明确终态结构
- 再按域逐步迁移

这比一次性全盘搬家稳得多。

## 9. 分阶段重构方案

## 阶段 0：冻结目标和原则

先不动代码，只统一原则：

1. 新代码优先往 `game/` 目标结构靠
2. 不再继续往 `app/api`、`app/modules`、`app/core` 的旧职责缝隙里随意加新文件
3. `core` 不再接收新的强游戏语义文件

这是第一步，不写代码也有价值。

## 阶段 1：先抽“目录外壳”，不搬大逻辑

先创建新目录：

```text
app/core/
app/game/
app/test_support/
```

其中：

- `app/core/` 可先保留原目录，但开始按子目录整理新文件
- `app/game/` 新建
- `app/test_support/` 新建

这一阶段重点是“搭新房”，不是“当天搬完所有家具”。

## 阶段 2：先迁 API 层和 schema 层

优先迁移这些低风险内容：

- `app/modules/account/AccountApi.py` -> `app/game/api/AuthApi.py`
- `app/api/GameBase.py` -> `app/game/api/BaseApi.py`
- `app/modules/task/TaskApi.py` -> `app/game/api/TaskApi.py`
- 其他 `XxxApi.py` 逐步迁移到 `app/game/api/`

同时迁移 schema：

- `app/schemas/Auth.py` -> `app/game/schemas/AuthSchema.py`
- `app/schemas/Game.py` 按领域拆到多个 schema 文件

为什么先迁这些：

- 改动可控
- 文件职责明确
- 对外接口路径可以不变
- 比直接动底层领域对象风险低

## 阶段 3：迁移游戏应用层

这一阶段是关键。

建议优先迁：

- `core/Dependencies.py` -> `game/application/Dependencies.py`
- `GameContext` -> `game/application/GameContext.py`
- `DailyReset.py` -> `game/application/DailyReset.py`
- `InitPlayerInfo.py` -> `game/application/InitPlayerInfo.py`

这是因为它们本质上都属于“游戏应用编排”，不是共享 core。

完成这一层后，目录语义会明显清晰很多。

## 阶段 4：迁移领域模块

按模块逐个迁移：

- `modules/task/*` -> `game/domain/task/*`
- `modules/mail/*` -> `game/domain/mail/*`
- `modules/spell/*` -> `game/domain/spell/*`
- `modules/inventory/*` -> `game/domain/inventory/*`
- 其他模块同理

建议迁移顺序：

1. `task`
2. `mail`
3. `inventory`
4. `spell`
5. `cultivation`
6. `alchemy`
7. `lianli`
8. `herb`
9. `account`
10. `player`

原因：

- 前几个耦合面相对小
- `player/cultivation/alchemy/lianli/herb` 的运行态联动更复杂，放后面更稳

## 阶段 5：抽离静态内容

把各模块里的 `json` 文件统一迁到 `game/content/`。

这样能解决两个问题：

- 领域代码目录更干净
- 游戏内容资源不再散落在各个模块文件夹

## 阶段 6：单独安置测试支持逻辑

建议处理：

- `app/api/Test.py` -> `app/test_support/api/TestApi.py`
- `core/TestAccountSeeder.py` -> `app/test_support/bootstrap/TestAccountSeeder.py`

这样正式运行逻辑和测试辅助逻辑才能真正分开。

第一轮实际也已经完成了这一步。

## 10. 迁移中的关键策略

## 10.1 不改接口路径，只改文件归属

第一轮迁移时，尽量保持这些对外路径不变：

- `/api/auth/*`
- `/api/game/*`
- `/api/test/*`

这样客户端和测试不用跟着一起改。

也就是说，先改“代码住哪”，后改“路径叫什么”。

## 10.2 每次只迁一层，不多层联动

最忌讳的做法是一次同时：

- 改目录
- 改命名
- 改业务逻辑
- 改 schema
- 改测试

这会让问题定位极难。

推荐原则：

- 一次只做一类迁移
- 迁完就跑测试

## 10.3 通过兼容导入降低风险

在迁移过程中，可以短期保留兼容层，但默认目标是后续继续删除，而不是长期维持。

例如：

- 旧文件保留，但只转发 import
- 新文件成为真实实现

这样可以分阶段替换引用，不用一口气改完所有 import。

## 10.4 先收口新代码入口

哪怕旧代码还没全部迁完，也应该先规定：

- 新增游戏 API 一律进入 `game/api`
- `core` 不再继续塞新业务文件

这能防止“重构还没开始，旧结构继续长新代码”。

## 12. 测试相关改动梳理

第一轮落地后，测试链路除了“通过”之外，还额外暴露过一个很实际的问题：

- 当多个测试进程或多个长链路测试共享同一个固定测试账号时，容易因为 `logout`、防作弊踢下线、或同账号状态重置而出现 `KICKED_OUT`

为降低这个问题，当前实现已经补了两层保护：

1. `/test/*` 不再只允许固定单一测试账号，而是允许：
   - 固定测试账号集合
   - `test_` 前缀的进程隔离测试账号
2. `pytest` 默认测试账号不再强依赖固定 `test`，而是优先使用：
   - `IDLE_TEST_USERNAME`
   - 若未设置，则自动生成 `test_<pid>`

另外，独立集成测试脚本也已切到单独账号命名，避免和常规 `pytest` 流程争用同一账号。

这套改动不是业务协议改动，而是测试基础设施改进，目的就是降低结构重构之后的回归噪音。

这轮继续推进 `core/security` 下沉时，又额外补了一层稳定性处理：

1. `TestApiClient` 的默认 `base_url` 改为运行时读取 `IDLE_TEST_BASE_URL`
2. `TestApiClient` 对瞬时 `502/503/504` 与空响应 JSON 解析失败增加了轻量重试
3. 在需要长时间全量回归时，建议单独起一台无 `--reload` 的测试服，例如：
   - `./venv/bin/python -m uvicorn app.main:app --host 127.0.0.1 --port 8445`
   - 再用 `IDLE_TEST_BASE_URL=http://127.0.0.1:8445/api` 跑测试

这样可以把“热更中的短暂不可用”与“真实业务回归失败”分开看，减少误报。

## 11.5 接口路径策略与客户端影响

这轮重构需要特别强调一个原则：

- 第一阶段优先重构“代码目录与内部依赖关系”
- 不优先改“对外 API 路径”

也就是说，这轮 `app` 重构首先是一次“内部结构重组”，不是一次“接口契约改版”。

### 为什么第一阶段不建议改接口路径

如果在第一阶段同时改：

- 目录结构
- import 路径
- 路由注册
- API URL
- 客户端请求地址

那就会把两类风险叠在一起：

1. 服务端内部结构重构风险
2. 前后端接口契约变更风险

这样不利于定位问题，也会明显增加回归成本。

### 第一阶段建议保持不变的内容

建议尽量保持这些对外路径不变：

- `/api/auth/*`
- `/api/game/*`
- `/api/test/*`

也就是说，即使后端代码从：

- `app/modules/task/TaskApi.py`

迁到了：

- `app/game/api/TaskApi.py`

对客户端来说，仍然继续访问原来的：

- `/api/game/task/list`
- `/api/game/task/claim`

### 客户端在第一阶段是否需要改动

如果第一阶段不改 API 路径，也不改请求/响应结构，那么客户端通常：

- 不需要改业务调用代码
- 不需要改接口路径常量
- 不需要改模块行为逻辑

客户端这边主要要做的是：

- 跑全量回归测试
- 验证所有现有接口行为未被目录重构破坏

### 哪些情况下客户端才需要改

只有出现以下情况，客户端才需要同步修改：

1. 服务端改了接口 URL
2. 服务端改了请求参数结构
3. 服务端改了响应字段结构
4. 服务端改了错误码或 reason_code 语义
5. 服务端改了 `/api/test/*` 相关路径或行为

### 对这轮重构的建议

建议把服务端重构拆成两类任务：

#### A. 第一轮：目录与内部结构重构

目标：

- 改目录
- 改 import
- 改内部组织
- 不改 API URL
- 不改响应契约

收益：

- 客户端基本无需改代码
- 风险集中在服务端内部

#### B. 第二轮：接口契约整理（如未来需要）

如果后续真的觉得某些路径命名不理想，再单独开一轮：

- 路径命名整理
- 客户端联动调整
- 文档同步
- 测试同步

不要把这件事和第一轮目录重构绑在一起做。

这轮重构如果真正开始做，测试绝对不是“最后顺手修一下”就够了，应该从第一阶段就跟着设计。

## 12.1 当前会受影响的测试类型

结合现有项目结构，至少会影响这些测试面：

1. 服务端 API 集成测试
2. 依赖导入类测试
3. 启动链路测试
4. 客户端依赖真实服务端的 API 化测试
5. 测试辅助接口与测试播种逻辑

这里最需要注意的是第 4 类。

因为你客户端已经有大量 GUT 在直接打本地服务端接口，所以服务端目录重构虽然“不改接口路径”，但如果重构过程中把启动、依赖注入、测试接口、测试账号播种搞坏，客户端测试会立刻跟着炸。

## 12.4 当前已落地的测试隔离改进

第一轮落地后，真实回归里出现过一次很典型的噪音问题：

- 多条测试链路共用同一测试账号时，容易因为 `logout`、防作弊踢下线、状态重置或并发回归而出现 `KICKED_OUT`

这不是业务协议设计错误，而是测试基础设施之前过于依赖固定账号。

目前已经补了这些隔离措施：

1. `/test/*` 不再只认可固定单一测试账号
   - 允许固定测试账号集合
   - 允许 `test_` 前缀的进程隔离测试账号
2. `pytest` 默认测试账号不再固定写死 `test`
   - 优先读取 `IDLE_TEST_USERNAME`
   - 未设置时自动生成 `test_<pid>`
3. 同一 pytest 进程内新建 `TestApiClient()` 时，也会沿用同一只隔离测试账号
4. 独立集成测试链路也改成使用隔离测试账号，避免与常规 `pytest` 流程争用

这套调整已经通过验证，目标就是把“结构重构带来的真实问题”和“测试账号争用造成的噪音”区分开。

## 12.6 兼容层的当前定位

测试层面最稳的策略不是“改完目录，一次性全切”，而是先把兼容层当作短期过渡：

- 新目录先成为真实实现
- 旧路径短期保留兼容 import

例如旧文件里只保留一层转发：

```python
from app.game.application.dependencies import *
```

这样做的好处：

- 服务端代码能逐步迁
- 测试不用同一天全改
- 出问题时更容易定位

但现在的方向已经明确：

- 兼容层是临时过渡手段
- 后续改造稳定后，可以直接删除旧路径代码
- 不需要为了长期保留旧路径，再额外增加专门的“防回退护栏”

## 14. 风险与收益评估

## 13.1 收益

- 目录职责更清晰
- `core` 从大杂烩回到共享层

- 如果一次改太多，测试回归成本高

## 13.3 结论

这是一轮值得做的重构，但必须按阶段推进，不能一把梭。

## 15. 最终建议

我的建议是：

把服务端 `app` 的当前重构目标明确设为：

```text
app/
  core/
  game/
  test_support/
```

然后按下面的优先级做：

1. 先冻结新的目录规范
2. 再迁 API 与 schema
3. 再迁游戏 application 层
4. 再迁 domain 模块
5. 最后处理测试支持目录拆分

如果只用一句话总结这份方案，那就是：

不要继续在现有 `api/modules/core` 这套历史结构上横向长更多东西，而是先把服务端收敛成“共享层 + 游戏域 + 测试支持”的结构，再用分阶段迁移把风险压住。后续如果要接运维系统，再在这个更干净的基础上单独引入 `app/ops`。
