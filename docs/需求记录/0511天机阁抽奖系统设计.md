# 天机阁抽奖系统设计

## 1. 目标与范围

本系统用于在「地区」页新增一个符合云稷城氛围的仙晶消耗入口，提供服务端权威结算的抽奖玩法。

一期只做一个常驻基础池：

- 消耗货币：`仙晶`
- 单抽：`3 仙晶`
- 十连：`30 仙晶`，实际产出 `10 + 2` 次奖励
- 奖励内容：术法解锁道具、筑基丹、聚气丹、空白玉简等物品
- 结算方式：服务端扣费、抽取、保底、发奖、返回结果，客户端只展示过程和结果

本期不做活动限定池、充值入口、复杂卡池轮换、道具兑换商店。抽奖配置全部放在服务端，客户端只消费接口返回。

---

## 2. 命名与地区定位

### 2.1 主推荐名称：天机阁

入口名称建议：`天机阁`

理由：

- 比「赌坊」更贴近修仙题材，像是以仙晶窥探天机、换取机缘。
- 可以自然解释概率抽取与保底机制：天机难测，但积累机缘终有显化。
- 一期只保留一个常驻池，后续如需扩展限定池再另开设计。

### 2.2 页面副标题与玩法名

- 建筑入口：`天机阁`
- 页面标题：`天机阁`
- 抽奖池名称：`云稷机缘`
- 抽奖动作文案：
  - 单抽：`祈愿一次`
  - 十连：`祈愿十次`

### 2.3 备选名称

- `万宝阁`：更像商店，适合后续兑换/拍卖系统。
- `灵契阁`：更偏契约与召唤，若未来抽伙伴/灵兽更合适。
- `问天台`：仪式感强，但作为地区建筑不如「阁」自然。

一期确定使用 `天机阁`。

## 3. 地区页入口设计

### 3.1 所属地区

入口放在「地区 Tab -> 云稷城」分组下，作为云稷城内的功能建筑。

建议排序：

1. `仙务司`
2. `炼丹坊`
3. `天机阁`（替换原「赌坊」入口）
4. 其他后续城市建筑

实现时直接删除/废弃当前「赌坊」地区卡片，使用 `天机阁` 替换，不保留赌坊入口。

### 3.2 区域卡片文案建议

标题：`天机阁`

图片区大字：`机`

简介：

```text
云稷城中观星问命之处，可消耗仙晶换取机缘，术法残卷与珍稀丹药皆有可能现世。
```

标签：

- `仙晶祈愿`
- `保底机缘`
- `常驻奖池`

按钮：

- 仙晶数量充足：`进入天机阁`
- 未解锁仙晶或数量为 0：仍可进入查看奖池，不建议在入口处置灰

说明：

- 抽奖入口不建议因仙晶不足而锁住，否则玩家无法查看概率、保底和奖励预览。
- 真正的仙晶不足提示应发生在页面内点击抽取时。

## 4. 抽奖规则

### 4.1 消耗规则

- 单抽消耗：`3 immortal_crystal`
- 十连消耗：`30 immortal_crystal`
- 十连实际奖励次数：`12`
- 十连的 `+2` 次奖励与普通抽取同池、同概率、同保底计数。

### 4.2 保底规则

推荐一期规则：

- 最高稀有度定义：`rarity = tian`
- 每 `60` 抽内必定获得至少 1 个 `rarity = tian` 奖励
- 抽到任意 `rarity = tian` 奖励后，保底计数重置为 `0`
- 如果第 `60` 抽仍未自然出 `rarity = tian`，该抽强制从 `rarity = tian` 奖池中随机一个奖励
- 十连 `10 + 2` 的 12 个结果逐个结算、逐个推进保底

`60` 抽作为一期定稿保底阈值。理由：

- 单抽 3 仙晶，则最差保底成本为 180 仙晶。
- 十连按 12 次计数，则最差大约 5 次十连触发保底。
- 对早期测试和体验来说，能较快看到高稀有奖励，不至于系统上线后长期无反馈。

如果未来 更高稀有度 出现，再单独评估保底是否升级为最高配置稀有度；一期固定保 `rarity = tian`。

### 4.3 概率建议

一期可先按物品 `rarity` 做稀有度层级：

| 稀有度 | 物品 rarity | 建议概率 |
| --- | --- | --- |
| 凡 | `fan` | 45% |
| 黄 | `huang` | 32% |
| 玄 | `xuan` | 16% |
| 地 | `di` | 5% |
| 天 | `tian` | 2% |

说明：

- 概率先按层级抽，再在该层级奖池内按权重抽具体物品。
- 相同稀有度内不同物品也必须配置独立 `weight`，初期可以全部设置为相同权重，方便后续运维人员直接手动调整。
- 保底只影响最终必须出 `rarity = tian` 的那次，不改变平时概率展示。

### 4.4 首抽特殊道具池

一期新增账号独立的首次祈愿特殊机制：

- 每个账号第一次在天机阁祈愿时，服务端内部必定命中特殊道具池中的 1 个道具。
- 特殊道具池同样由服务端配置，不写死在客户端。
- 初期配置默认放入 3 个主动天阶术法解锁道具。
- 首抽特殊道具只触发一次，触发后记录 `first_draw_special_claimed = true`。
- 首抽特殊道具与普通抽奖结果一起返回并发放到储纳。
- 如果玩家第一次操作是单抽，则本次返回 1 个特殊结果。
- 如果玩家第一次操作是十连，则先结算 1 次特殊池结果，再结算 11 次普通池结果，总结果仍为 12 个。
- 首抽特殊结果不计入 60 抽最高稀有度保底，也不重置普通池保底。
- 这是隐藏的新账号体验规则，不在客户端任意位置提示玩家，不在概率问号、不在奖池预览、不在结果角标、不在接口展示文案中暴露。玩家只会看到正常抽奖结果，误以为自己运气好即可。

### 4.5 重复术法道具

一期建议：

- 奖池不按玩家已学术法动态移除奖励。
- 抽到已学会术法的解锁道具，仍然发放到储纳中。
- 重复术法道具不转化为其他资源，保留为真实道具。后续术法升星系统可消耗多余术法道具。

理由：

- 服务端奖池更稳定，概率公示简单。
- 不会因为玩家进度不同导致概率难解释。
- 不需要在抽奖结果里额外显示「已拥有」标记。
- 不限制基础术法进入奖池，基础术法道具也可以作为后续升星材料。

## 6. 服务端设计

### 6.1 服务端职责

服务端必须作为唯一真值：

- 校验玩家是否登录、是否可抽取
- 校验仙晶数量是否足够
- 原子扣除仙晶
- 根据奖池配置和玩家保底状态抽奖
- 更新保底计数
- 发放物品到玩家储纳
- 返回本次抽奖结果与最新货币/保底状态
- 记录抽奖日志，方便后续排查争议

客户端不得本地决定抽奖结果。

### 6.2 数据结构建议

玩家侧新增：

```text
gacha_state:
  first_draw_special_claimed: bool
  pools:
    cloud_fate:
      pity_count: int
      total_draw_count: int
      last_draw_at: datetime
```

抽奖历史可单独存储，避免玩家主档膨胀：

```text
gacha_history:
  id
  player_id
  pool_id
  draw_type
  cost
  results
  pity_before
  pity_after
  first_draw_special_triggered
  created_at
```

### 6.3 接口建议

#### GET `/api/game/gacha/pool`

用途：进入天机阁时拉取奖池展示信息。

返回建议：

- 当前池子 ID、名称、描述
- 单抽消耗、十连消耗、十连奖励次数
- 保底阈值、当前保底计数、距离保底还差几抽
- 概率公示
- 奖励预览列表
- 玩家当前仙晶数量

#### POST `/api/game/gacha/draw`

请求：

```json
{
  "pool_id": "cloud_fate",
  "draw_type": "single | multi",
  "client_request_id": "uuid"
}
```

返回：

```json
{
  "success": true,
  "pool_id": "cloud_fate",
  "draw_type": "multi",
  "cost": {"immortal_crystal": 30},
  "results": [
    {
      "item_id": "spell_fireball",
      "count": 1,
      "rarity": "huang",
      "is_pity": false
    }
  ],
  "currency": {
    "immortal_crystal": 120
  },
  "pity": {
    "threshold": 60,
    "current": 12,
    "remaining": 48
  }
}
```

失败原因建议使用 reason code：

- `NOT_ENOUGH_IMMORTAL_CRYSTAL`
- `POOL_NOT_FOUND`
- `POOL_CLOSED`
- `INVALID_DRAW_TYPE`
- `REQUEST_DUPLICATED`

### 6.4 配置归属

抽奖配置全部在服务端维护，建议至少包含：

- 池子基础配置：`pool_id`、名称、描述、是否开放
- 消耗配置：单抽、十连、十连奖励次数
- 稀有度概率配置
- 具体奖励配置：`item_id`、`count`、`rarity`、`weight`
- 保底配置：阈值、目标稀有度
- 首抽特殊池配置：候选 `item_id`、`count`、`weight`。该配置仅服务端内部使用，不通过客户端展示接口暴露。
- 概率说明文案或公式展示数据

客户端只展示服务端返回，不应本地硬编码真实概率和奖池内容。

### 6.5 幂等与并发

抽奖接口必须支持幂等：

- 客户端每次点击生成 `client_request_id`
- 服务端同一玩家同一 `client_request_id` 只结算一次
- 如果客户端超时重试，服务端返回首次结算结果，不重复扣费

并发保护：

- 抽奖结算必须在玩家写锁/事务中完成
- 扣仙晶、抽奖励、更新保底、发物品要么全部成功，要么全部失败

## 7. 客户端 UI 与交互

### 7.1 页面结构

进入 `天机阁` 后展示独立页面：

- 顶部：
  - 返回按钮
  - 标题：`天机阁`
  - 当前仙晶数量
- 池子区域：
  - 池名：`云稷机缘`
  - 简介
  - 保底进度：`距离传说机缘还差 X 抽`
  - 问号图标：点击打开概率公式/概率详情弹窗，展示内容来自服务端配置
- 奖励预览：
  - 按稀有度分组展示
  - 可点击查看物品详情
- 底部操作：
  - `祈愿一次 3仙晶`
  - `祈愿十次 30仙晶（10+2）`

### 7.2 仙晶不足

- 按钮可以保持可点击，点击后由客户端根据当前展示值先做轻提示。
- 如果客户端本地值过期，仍以服务端失败返回为准。
- 提示文案：

```text
仙晶不足，无法祈愿。
```

### 7.3 抽奖结果展示

单抽：

- 播放短动画
- 显示一个结果卡
- 高稀有度结果必须使用特殊动效、震动与音效反馈，至少区分 `rarity = di` 和 `rarity = tian`

十连：

- 播放阵法展开动画
- 结果以 `3 x 4` 或纵向卡片列表展示 12 个奖励
- 支持跳过动画
- 结果页按钮：
  - `再祈愿十次`
  - `收下`

### 7.4 抽奖反馈方向

颜色必须复用物品稀有度色体系：`rarity = di` 使用紫色，`rarity = tian` 使用橙色，避免术法/储纳/抽奖三个系统对同一稀有度出现不同颜色。

抽奖反馈建议走修仙题材，而不是现代抽卡机。高稀有结果需要形成「动效 + 震动 + 音效」的组合反馈：

1. 仙晶飞入阵眼
2. 云纹阵盘点亮
3. 星砂/符文环绕
4. 按稀有度变色：
   - 普通：白/灰光，短闪后直接出卡
   - 精良：青光，轻微符文闪烁
   - 稀有：蓝紫光，阵盘短暂旋转
   - 史诗：紫光，对齐现有 `rarity = di` 物品颜色 `#EE82EE`，阵盘停顿蓄光，结果卡翻出时带紫色流光，配短促轻震和清脆法铃音效
   - 传说：橙光，对齐现有 `rarity = tian` 物品颜色 `Color.ORANGE`，短暂暗屏聚焦、屏幕轻震、橙色符文爆开后再出卡，配更明显震动和低频钟鸣/灵爆音效
5. 结果卡从光中翻出

反馈约束：

- 动画应可跳过；如果一次十连中存在高稀有度结果，跳过后直接进入结果汇总，不强制播放完整高稀有动画。
- 震动只在移动端启用，并尊重系统震动能力、游戏内音效/震动开关；桌面端不做震动。
- 音效必须尊重游戏内静音/音效开关，静音时不播放。
- 移动端避免大量实时粒子叠加，优先用少量 TextureRect、Tween、Shader/渐变背景实现。
- 十连动画不应阻塞服务端请求；流程应为「请求服务端 -> 返回结果 -> 播放已知结果动画」。

## 8. 与现有系统的关系

### 8.1 储纳

- 抽奖成功后，服务端直接发放物品到储纳。
- 客户端收到结果后可立即展示结果卡。
- 随后刷新本地玩家数据或局部更新储纳数据，保证进入储纳页时数量正确。

建议手感：

- 抽奖结果弹窗出现时，不等待全量刷新 UI。
- 服务端返回结果后先展示奖励，再异步刷新玩家数据。

### 8.2 货币栏

- 抽奖成功后需要立即更新右上角仙晶数量。
- 服务端返回中应包含最新 `immortal_crystal` 数量，客户端无需等待全量刷新才扣数显示。

### 8.3 日志

抽奖结果可以写入系统日志，但不要刷屏：

- 单抽：记录完整奖励。
- 十连：记录摘要，例如 `天机阁祈愿获得 12 件物品，其中传说 1 件、史诗 0 件`。
- 详细结果只在抽奖结果页展示。

## 9. 测试计划

### 9.1 服务端测试

必须覆盖：

- 单抽扣除 3 仙晶并发奖
- 十连扣除 30 仙晶并发 12 个奖励
- 仙晶不足不扣费、不发奖
- 保底计数递增
- 抽到最高稀有度后保底清零
- 到达 60 抽保底阈值时强制出最高稀有度
- 十连中途触发保底后正确清零并继续计数
- 账号第一次抽奖服务端内部必定命中首抽特殊道具池，且只触发一次
- 首抽特殊结果不计入普通 60 抽保底，不重置普通池保底
- 幂等请求不会重复扣费
- 并发抽奖不会造成仙晶负数或重复发奖

### 9.2 客户端测试

必须覆盖：

- 地区页展示 `天机阁` 卡片，原赌坊入口不再显示
- 进入天机阁后展示仙晶数量、单抽/十连价格、保底进度、问号概率入口
- 仙晶不足提示
- 单抽/十连成功后展示结果
- 右上角仙晶数量即时刷新
- 抽奖结果进入储纳后数量正确
- 动画跳过按钮可用

### 9.3 联调测试

重点关注：

- 服务端返回速度正常但客户端动画/刷新不应造成体感卡顿
- 十连 12 个结果同时展示时不应掉帧明显

## 11. 一期建议结论

一期推荐按以下口径落地：

- 名称：`天机阁`
- 池子：`云稷机缘`
- 单抽：`3 仙晶`
