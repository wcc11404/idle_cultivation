# 0429 线上可视化运维系统设计（结合当前项目结构重整理版）

## 1. 文档目的

这份文档用于明确“一期线上可视化运维系统”在当前项目结构下应该如何落地，重点回答这些问题：

1. 这是不是前后端分层方案
2. 前后端分别用什么语言和框架
3. 在当前服务端真实目录下，代码应该放在哪里
4. 现有 `/api/admin/*`、`admin_support`、`bootstrap` 与未来运维系统是什么关系
5. 一期先做哪些，哪些暂时不做

这次重整理的目标，不是重新发明一套方案，而是把原设计稿和当前真实代码结构对齐，让它能直接指导后续开发。

---

## 2. 结论先说

### 2.1 是否前后端分层

是，但属于“同仓库、同机部署”的前后端分层方案，不建议一期拆成两套完全独立的仓库。

更准确地说：

- 前端：单独一个运维网页
- 后端：单独一层 `/ops/api/*` 运维接口
- 部署：仍和现有游戏服务端放在同一台机器上
- 代码：建议都放在 `idle_cultivation_server` 仓库里维护

这意味着：

- 前端和后端职责分离
- 运维前端不直接碰数据库
- 运维前端不直接调用 `/api/game/*`
- 运维前端也不应长期直接调用当前 `/api/admin/*`
- 所有正式运维操作最终统一经过 `/ops/api/*`

### 2.2 推荐技术选型

结合当前项目现状，一期推荐技术栈如下：

- 运维后端：`Python + FastAPI`
- 运维前端：`TypeScript + React + Vite`
- 静态资源托管：一期可先用 `FastAPI StaticFiles`，后续再接 `Nginx`
- 数据库存储：沿用当前服务端数据库 `PostgreSQL`

推荐原因：

- 现有游戏服务端已经是 `FastAPI`
- 当前服务端已经完成目录重构，具备比较清晰的分层基础
- 运维网页需要表格、筛选、批量操作、确认流，`React + Vite` 更适合长期维护
- 一期不建议用 Godot 做运维工具页
- 一期也不建议直接用 Streamlit 作为正式方案，虽然开发快，但后续权限、审计、复杂交互会比较吃力

## 3. 一期目标与边界

### 3.1 系统目标

为线上环境提供一套“网页登录可视化运维系统”，让有权限的管理员可以通过浏览器完成以下工作：

- 查看服务状态
- 查看玩家列表与玩家详情
- 封禁 / 解封玩家
- 强制玩家下线
- 单发 / 群发物品
- 单发 / 群发邮件
- 查看关键操作审计日志
- 管理禁止登录状态

### 3.2 一期必须做到

- 可登录
- 可鉴权
- 可做权限校验
- 可做高危操作确认
- 可记录审计日志
- 可区分成功与失败原因
- 支持全服群发且入口足够直接
- 只允许发物品、发邮件，不允许直接改资源数值
- 只做永久封禁，但必须支持解除封禁
- 禁止登录时保留白名单测试账号

### 3.3 一期暂不做

- BI 报表
- 复杂图表大盘
- 自动活动编排
- 多角色复杂审批流
- 短信 / OTP 二次验证
- 完整在线行为分析
- 多环境统一总控台
- 真正执行服务器重启
- 为运维系统单独推动 Nginx 升级改造

## 4. 当前项目基础与现实约束

### 4.1 当前服务端真实结构

当前 `idle_cultivation_server/app` 已经不是旧的 `api + modules + schemas` 结构，而是：

```text
app/
  admin_support/
  bootstrap/
  core/
  game/
  test_support/
  main.py
```

其中：

- `app/bootstrap/ApiRouter.py` 负责顶层路由装配
- `app/game/*` 是游戏正式主线
- `app/core/*` 是共享基础设施
- `app/test_support/*` 承接 `/api/test/*`
- `app/admin_support/api/AdminApi.py` 承接当前仍存在的 `/api/admin/*`

这意味着后续设计运维系统时，不能再按“旧 `app/api/Admin.py` + `app/modules/*`”的脑图去规划，而应该以现在这套真实结构为基础继续扩展。

### 4.2 当前对外路径现状

虽然服务端内部结构已经重构，但当前对外接口路径仍保持兼容：

- `/api/auth/*`
- `/api/game/*`
- `/api/test/*`
- `/api/admin/*`

当前 `app/bootstrap/ApiRouter.py` 的真实装配关系是：

```text
app/main.py
  -> app/bootstrap/ApiRouter.py
     -> app/game/api
     -> app/test_support/api
     -> app/admin_support/api
```

所以：

- `/api/admin/*` 现在仍是真实可用入口
- 但它更适合被视为“历史管理接口承载层”
- 不建议把它继续扩张成正式长期运维平台

### 4.3 当前已有基础

当前项目已经具备以下落地条件：

- 服务端技术栈是 `FastAPI`
- 已有共享配置、安全、数据库、写锁、资源文件等基础设施
- 已存在一批早期管理能力，当前真值位于 `app/admin_support/api/AdminApi.py`
- 已具备部分管理动作，例如：
  - 管理员登录
  - 玩家列表
  - 玩家详情
  - 封号
  - 发邮件

因此，一期不应推倒重做，而应在现有服务端基础上把这些早期能力迁移并收敛到正式的 `ops` 域里。

### 4.4 当前明显不足

现有 `admin_support` 更像“早期管理接口承载层”，还不是正式运维平台，主要问题包括：

- 管理员账号目前偏临时实现
- 权限模型不完整
- 审计体系缺失
- 高危操作缺少二次确认
- 前端操作流尚不存在
- 批量能力不完整
- 系统级维护能力尚未正式收口

所以一期的核心，不是简单“再写个网页”，而是把这套能力治理起来。

## 5. 总体架构

### 5.1 一期推荐总结构

基于当前服务端真实结构，一期推荐把正式运维系统补成这样：

```text
idle_cultivation_server/
  app/
    admin_support/
    bootstrap/
    core/
    game/
    test_support/
    ops/
      api/
      services/
      auth/
      audit/
      models/
  ops_web/
    src/
    package.json
    vite.config.ts
```

这里有两个关键点：

- `ops` 是在当前真实结构上新增的正式运维域
- `admin_support` 不是最终方向，而是后续要逐步被 `ops` 吸收或替代的历史承载层

### 5.2 调用关系

标准链路如下：

`Browser -> ops_web -> /ops/api/* -> app/ops/services/* -> game/core/db`

重点约束：

- 运维前端只调 `/ops/api/*`
- 运维前端不直接调 `/api/game/*`
- 运维前端不直接调当前 `/api/admin/*`
- 运维前端不直接连数据库
- 运维接口不能假设“只有前端页面会访问”，必须始终做鉴权与权限校验

### 5.3 当前 `admin_support` 与未来 `ops` 的关系

这块必须说清楚，不然后面开发时很容易反复横跳。

当前判断如下：

- `app/admin_support`：现有历史管理接口承载层
- `app/ops`：未来正式运维平台域

建议策略：

- 一期开发时，不再继续扩张 `/api/admin/*`
- 现有 `AdminApi` 中已经验证可用的逻辑，可以迁移或下沉到 `ops/services/*`
- 新增运维能力一律优先落到 `app/ops/*`
- 当前 `/api/admin/*` 可以保留一段迁移期，但不再作为长期正式入口设计

一句话说：

`admin_support` 是当前现实，`ops` 才是正式未来。

## 6. 为什么这样分层

### 6.1 和当前真实结构兼容

现在服务端已经有明确的：

- `bootstrap`
- `core`
- `game`
- `test_support`
- `admin_support`

所以正式运维系统最自然的做法，不是回头再造一套平行旧结构，而是在它旁边新增 `ops` 域。

### 6.2 安全边界更清晰

如果继续把正式运维能力堆进 `/api/admin/*`，后面会越来越难区分：

- 哪些是历史临时接口
- 哪些是正式后台能力
- 哪些做了审计
- 哪些做了确认流

统一收口到 `/ops/api/*` 后：

- 权限统一
- 审计统一
- 错误格式统一
- 确认流程统一

### 6.3 复用已有能力，避免重复造轮子

现有 `AdminApi` 已经做了一部分能力，一期最好迁移并复用已有实现思路，而不是再并行维护两套正式后台接口。

### 6.4 便于长期演进

以后如果要加：

- 多角色权限
- 审批流
- 风险拦截
- 批量模板
- 自动化脚本
- 更完整的后台页面

都可以继续长在 `app/ops/*` 这一层，而不是把 `admin_support` 继续越堆越重。

## 7. 一期技术选型说明

### 7.1 运维后端

推荐：

- 语言：`Python`
- 框架：`FastAPI`

原因：

- 和现有服务端同技术栈
- 可直接复用当前 `app/core/*` 下的配置、安全、数据库初始化、日志体系
- 可与当前项目部署方式保持一致
- 可自然接入当前 `bootstrap` 路由装配方式

### 7.2 运维前端

推荐：

- 语言：`TypeScript`
- 框架：`React`
- 工程工具：`Vite`
- UI 组件：一期可先轻量自写，或后续补组件库

原因：

- 表格、表单、筛选、分页、批量选择都比较成熟
- 开发体验和后续维护性较好
- 页面组织比服务端模板渲染更清晰
- 比 Streamlit 更适合做正式运维后台

### 7.3 为什么不建议一期用服务端模板

服务端模板渲染不是不能做，但不太适合这个项目的一期方向，主要原因：

- 后台会有较多交互状态
- 批量操作、二次确认、分页筛选会更繁琐
- 后面演进为正式后台时，模板页通常还得重构

### 7.4 为什么不建议运维前端直接放到客户端仓库

运维系统本质上服务于服务器运维，不属于游戏客户端的一部分。

所以建议：

- 设计稿可以写在客户端 `docs/需求记录/`
- 但真正代码落地放到 `idle_cultivation_server`

这样更符合职责边界。

## 9. 账号、鉴权与权限模型

### 9.1 运维账号

一期先支持一类账号：

- `super_admin`

它具备所有权限，但实现上仍建议按权限点建模，不要直接写死成“只有一个管理员所以不做权限”。

### 9.2 建议预留的权限点

- `player.view`
- `player.ban`
- `player.kick`
- `grant.item`
- `grant.mail`
- `system.view`
- `system.control`
- `audit.view`

### 9.3 鉴权建议

- 登录成功后返回运维专用 token
- 运维 token 和玩家 token 必须完全分离
- token 应有过期时间
- 服务端应支持主动失效 / 撤销
- `/ops/api/*` 是对外唯一正式运维入口

### 9.4 最低安全要求

- 密码必须哈希存储
- 登录失败做限流
- 高危操作需二次确认
- 所有写操作必须记审计
- 生产环境必须启用 HTTPS
- 审计日志保留时长：永久

### 10.7 禁止登录与维护入口

一期先做“入口与状态管理”，不执行真实重启动作。

支持：

- 开启禁止登录
- 关闭禁止登录
- 配置或查看白名单测试账号
- 预留“踢出所有在线玩家”入口
- 预留“重启服务器”入口

一期边界：

- “踢出所有在线玩家”可先不接真实动作，视开发节奏决定
- “重启服务器”只保留页面入口和后端占位，不调用任何实际功能
- 白名单测试账号在禁止登录期间仍可进入游戏

约束：

- 仅 `super_admin`
- 必须二次确认
- 必须有互斥锁
- 必须写审计

### 10.7.1 登录闸门 + 白名单实现方案

一期明确采用“服务端登录闸门”方案，不在客户端或运维前端做安全拦截。

核心原则：

- 是否允许登录，由游戏服务端登录接口最终判定
- 运维前端只负责配置开关和白名单，不负责真正放行
- 白名单账号在“禁止登录”开启时仍可登录
- 白名单判断按 `account_id` 进行，不按昵称判断

建议登录链路：

1. 客户端发起登录请求
2. 服务端校验账号密码
3. 服务端校验账号是否被永久封禁
4. 服务端读取 `ops_system_state.login_gate_enabled`
5. 若 `login_gate_enabled = false`，正常放行登录
6. 若 `login_gate_enabled = true`，继续检查 `ops_login_whitelist`
7. 若当前账号在白名单中，允许继续登录
8. 若当前账号不在白名单中，拒绝登录并返回明确业务错误码

建议错误码：

- `LOGIN_DISABLED`
- `LOGIN_DISABLED_NOT_IN_WHITELIST`

建议前端提示文案：

- `服务器维护中，当前账号未开放登录`

运维后台需要提供的能力：

- 开启禁止登录
- 关闭禁止登录
- 查看当前白名单
- 添加白名单账号
- 移除白名单账号
- 查看最近一次闸门状态变更人与时间

审计要求：

- 开启禁止登录必须记录审计
- 关闭禁止登录必须记录审计
- 白名单增删必须记录审计
- 白名单测试账号登录本身不必写审计日志，但白名单配置变更必须写审计

## 11. `ops/api` 与 `ops/services` 的职责划分

### 11.1 `ops/api` 的定位

`ops/api` 是正式运维接口层。

它负责：

- 接收前端请求
- 做鉴权与权限校验
- 做参数校验
- 做统一返回格式
- 调用内部服务层

### 11.2 `ops/services` 的定位

`ops/services` 是内部运维服务层。

它负责：

- 编排具体运维动作
- 复用游戏业务模块能力
- 复用当前 `admin_support` 中可迁移的已有实现思路
- 处理批量逻辑与高危确认流
- 调用审计与状态存储能力

### 11.3 一期原则

- 对外正式运维入口只保留 `/ops/api/*`
- 不再新增 `/api/admin/*` 能力

### 16.3 路由装配落点

基于当前真实结构，后续运维接口接入时，建议由：

- `app/bootstrap/ApiRouter.py`

统一装配：

- `app/game/api`
- `app/test_support/api`
- `app/admin_support/api`
- `app/ops/api`

这意味着：

- `ops` 的接入不需要重新发明一套启动方式
- 只是在现有 `bootstrap` 总装配层继续扩展

### 16.4 Nginx 统一入口（后续优化项）

示意：

- `/api/*` 转发到游戏服务
- `/ops/api/*` 转发到运维 API
- `/ops/*` 转发到运维前端静态页

说明：

- 这部分属于后续整体部署优化
- 一期文档保留设计方向即可，当前不要求立即落地

### 16.5 环境隔离

必须区分：

- 本地开发
- 测试环境
- 正式环境

禁止共用：

- 运维密码
- 运维 token
- 运维数据库配置

### 16.6 环境安装流程升级建议

当前服务端已经有一版可用的环境安装脚本：

- `idle_cultivation_server/setup_ubuntu.sh`

这版脚本当前已经能完成：

- 安装 `Python`、`venv`、`pip`
- 安装 `PostgreSQL`
- 创建服务端虚拟环境并安装 `requirements.txt`
- 初始化主游戏数据库 `idle_cultivation_game`
- 可选启动服务端

这说明一期运维系统不需要从零设计安装流程，而应该在现有脚本和现有运维文档的基础上升级。

建议把环境安装流程拆成两层：

#### A. 基础服务端环境层

继续由现有：

- `setup_ubuntu.sh`
- `docs/03-operations/安装与启动手册.md`

负责保证这些基础能力：

- Python 环境
- PostgreSQL 环境
- `venv` 健康检查与重建
- 主游戏数据库初始化
- 服务端启动与重启

#### B. 运维系统增量环境层

在正式开始做 `ops_web` 和 `app/ops/*` 后，安装流程需要新增这些内容：

- 安装 `Node.js`
- 安装 `npm`
- 安装前端依赖：`ops_web/package.json`
- 构建前端产物：`npm run build`
- 处理前端构建产物的托管目录
- 初始化运维系统新增表

建议的升级方向：

1. 继续保留 `setup_ubuntu.sh` 作为总入口
2. 新增运维系统相关参数或子步骤，例如：
   - `--with-ops-web`
   - `--skip-ops-web`
   - `--init-ops-db`
3. 在脚本内部把“主游戏环境初始化”和“运维系统增量初始化”明确拆开
4. 不要把所有逻辑都继续堆在一个无分层的大 shell 文件里，必要时可拆成：
   - `scripts/setup_python_env.sh`
   - `scripts/setup_postgres.sh`
   - `scripts/setup_ops_web.sh`
   - `scripts/init_ops_db.sh`

这样做的好处是：

- 兼容当前已有部署方式
- 不会推翻现有 `setup_ubuntu.sh` 的价值
- 后续增加 `ops_web` 构建步骤时更自然
- 新人接手时也更容易判断“我现在只缺哪一层环境”

### 16.7 一期环境安装文档要求

当运维系统开始进入开发时，除了代码本身，还必须同步补一套环境安装说明，不能只靠口头经验。

- 服务端与运维前端分别如何启动
- `ops_web` 如何构建与托管
- 哪些环境变量属于游戏主线，哪些属于运维系统

建议后续在服务端正式文档中单独补：

## 19. 风险点与建议

### 19.1 最大风险不是页面，而是高危写操作治理

这个系统最容易出事的地方不是“列表显示错了”，而是：

- 发错全服奖励
- 封错玩家
- 误踢全服
- 误重启

所以优先级应始终是：

安全 > 审计 > 权限 > 确认流 > 页面体验

### 19.2 一期不做真实重启动作

建议只做：

- 重启入口占位
- 后续实现预留

一期不做：

- 游戏服务进程重启
- 整台机器重启

原因是当前目标是先把运维入口、权限、审计、发放和玩家管控跑通，不必提前引入更高风险动作。

### 19.3 `Nginx` 不是一期前置条件

`Nginx` 统一入口是一个合理方向，但它属于整体部署体系优化，不只是运维系统自身升级。

所以一期可以先：

- 把运维后端和前端结构搭好
- 先用后端托管静态资源或简单部署方式跑通
- 后续再统一收口到 `Nginx`

### 19.4 一期不建议承诺强一致在线人数

如果当前还没有稳定在线会话中心，一期首页可以展示：

- 近实时活跃人数
- 最近在线玩家数

不要先把文案写成强一致“当前在线”。

### 19.5 一期不要直接开放太多管理员

建议先只开：

- `super_admin`

等系统稳定后，再扩展普通运维角色。

### 19.6 当前结构下的迁移风险点

由于当前真实服务端已经存在 `admin_support`：

- 一期开发时很容易图省事，继续在 `AdminApi.py` 上追加新能力
- 这样短期快，但长期会让 `ops` 域失去意义

所以需要明确约束：

- `admin_support` 可以参考、可以迁移、可以临时兼容
- 但不应再继续成为长期正式运维能力的主增长点

## 20. 已拍板项

1. 群发允许全服所有玩家，而且要做成清晰直接的正式入口

## 22. 最终建议

一期推荐方案可以概括为一句话：

在当前已经完成重构的 `idle_cultivation_server` 服务端结构上，新增一套 `app/ops/*` 的正式运维域，并配一个 `TypeScript + React + Vite` 的 `ops_web` 运维网页前端；前端只调 `/ops/api/*`，后端通过 `ops/api + ops/services + ops/auth + ops/audit + ops/models` 统一承接权限、确认、审计、发放、封禁/解封和禁止登录白名单能力。当前 `app/admin_support/api/AdminApi.py` 继续作为历史管理接口承载层存在，但不再作为正式长期方案扩张；顶层路由装配继续通过 `app/bootstrap/ApiRouter.py` 统一接入；重启功能先只保留入口占位，`Nginx` 统一入口留作后续整体部署优化。

如果后面按这个稿子推进，开发上会比较顺：

- 方向清楚
- 分层清楚
- 和当前真实结构不打架
- 历史 `admin_support` 的定位清楚
- 一期优先级也清楚

这份文档的目标，就是把这些边界先钉住。
