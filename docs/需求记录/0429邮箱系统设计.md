# 邮箱系统设计（一期草案）

## 1. 目标与范围

### 1.1 目标
- 提供服务端权威的“邮件收发奖励”能力，用于发放补偿、活动奖励、运营通知、系统通知。
- 客户端提供统一邮箱入口、列表浏览、详情查看、单封领取、单封删除、批量清理能力。

### 1.2 一期范围
- 客户端：
  - 设置页“邮箱”按钮进入邮箱页。
  - 邮件列表展示、邮件详情弹窗、领取/删除、批量删除已读且已领取。
- 服务端：
  - 邮件存储与状态管理。
  - 邮件列表、领取、删除、批量删除接口。
- 不做：
  - 实时推送（先走拉取）。
  - 附件拆分领取（一期默认整封领取）。
  - 邮件搜索/筛选/置顶。

## 2. 客户端 UI 设计

## 2.1 入口与页面结构
- 入口：`设置` Tab 内“邮箱”按钮。
- 打开后进入独立 `MailPanel`（风格对齐排行榜页顶部）。
- 顶部区域（固定）：
  - 左：返回按钮（复用炼丹坊/排行榜返回按钮样式）
  - 中：标题“邮箱”
  - 下：一条横向分割线

## 2.2 列表页布局
- 顶部信息行（在分割线下方）：
  - 左对齐：`邮件数量 / 容量`（例：`23 / 100`）
  - 右侧按钮：`一键删除`
- 默认容量上限：`100`
- 下方为邮件列表（仅纵向滚动）：
  - 禁止横向滚动
  - 隐藏横向/纵向滚动轴
  - 列表容器滚动行为对齐任务列表页
  - 每个邮件缩略卡可点击。
  - 缩略卡内容：
    - 第1行：邮件标题
    - 第2行：正文首行简要预览（超长省略）
    - 第3行：发送时间
    - 角标：`未读/已读` 标识（点击详情后变为已读）
  - 右侧独立列：
    - 若有附件：显示“第一个附件”的格子预览（复用储纳物品格样式）
    - 若无附件：留空
    - 若已领取：仍显示第一个附件，但渲染为“已领取态”（建议加灰化 + 小角标“已领”）

## 2.3 详情弹窗布局
- 以弹窗展示（复用现有弹窗模板体系）：
  - 大标题：邮件标题
  - 分割线
  - 发送时间
  - 正文区（固定高度，可纵向滚动）
  - 附件区（若有）：
    - 使用储纳格子风格展示附件
    - 每行最多4个，可换行（最多展示3行）
    - 附件最多10种不同物品（每种可堆叠）
  - 底部按钮：
    - `领取`（可领时启用，已领取后置灰或改“已领取”）
    - `删除`

## 2.4 移动端触摸交互优化（与任务页同策略）
- 邮件列表项使用“滑动/点击分离”策略：
  - 手指移动距离超过阈值（touch slop）判定为滚动，不触发点击。
  - 仅在按下-抬起位移小于阈值时才判定为点击卡片。
- 卡片内按钮/可点击区域优先触发自身点击，不吞掉外层纵向滚动。
- 目标：避免“手机上点卡片时把列表卡死”或“滚动被点击误触发”。

## 2.5 邮箱页视觉风格约束
- 保持游戏内现有配色体系，不新增跳脱主题色：
  - 背景主色沿用当前页面底色（`#eadab9` 体系）。
  - 卡片底色沿用 `#f2e5cc`。
  - 边框/分割线沿用现有暖棕描边。
  - 主操作按钮继续复用模板（黄/红/绿/灰）：
    - 领取可用：绿色
    - 领取不可用：灰色
    - 删除：红色
- 未读/已读标识与已领取角标采用低饱和、可读优先，不引入荧光色。

## 3. 客户端交互与状态规则

## 3.1 邮件状态（双维度）
- 维度A（阅读态）：
  - `未读`
  - `已读`
- 维度B（附件态）：
  - `无附件`
  - `未领取`
  - `已领取`

说明：
- 点击缩略卡打开详情，即标记为“已读”（服务端同步）。
- 有附件邮件领取后进入“已领取”。
- 无附件邮件可直接读后删除，也可保留。

## 3.2 操作行为
- 点击邮件卡：
  - 打开详情弹窗
  - 若此前未读，触发“标记已读”
- 点击领取：
  - 调服务端领取接口
  - 成功后刷新当前邮件状态 + 顶部货币/背包
- 点击删除：
  - 建议二次确认
  - 成功后从列表移除
- 点击“一键删除已读&已领取”：
  - 先弹确认弹窗（文案示例：`将批量清除所有已读且已领取附件的邮件，是否继续？`）
  - 仅清理满足条件邮件
  - 删除后刷新列表计数

## 3.4 登录态内的客户端统一定时刷新（红点前置能力）
- 客户端新增统一刷新函数（建议挂在 `GameUI`）：
  - 函数示例：`refresh_system_states_tick()`
  - 每 `X` 秒执行一次（一期建议 `10s`）。
- 串行刷新顺序（固定）：
  1. `GET /api/game/task/list`
  2. `GET /api/game/mail/list`
- 该刷新与页面解耦：
  - 用户不在任务页/邮箱页时也刷新状态，用于后续红点提示。
- 刷新结果只更新轻量状态缓存：
  - 任务可领取计数、邮件未读数、邮件总数等。
- 失败策略：
  - 单次失败静默，不打断当前交互。
  - 下个周期继续尝试。
- 生命周期建议：
  - 登录成功后启动；
  - 登出、鉴权失效、被踢后停止。

## 3.3 展示排序（建议）
- 默认按发送时间倒序（新到旧）。
- 同时间按邮件ID倒序兜底。
- 一期不区分已读/未读分组，保持时间线更直观。

## 4. 服务端设计

## 4.1 数据模型建议（锁定独立新表）
- 新增独立表 `mail_data`（不并入 `player_data.data`）：
  - `mail_id`（唯一ID）
  - `account_id`
  - `title`
  - `content`
  - `attachments`（`[{item_id, count}]`）
  - `created_at`（发送时间）
  - `expire_at`（可选，预留）
- `is_read`（bool）
- `is_claimed`（bool）
- `claimed_at`（可选）
- `is_deleted`（bool，软删）
- `deleted_at`（可选，软删时间）

备注：
- 本期直接采用独立表，方便后续运维人员通过 SQL / 运维脚本发邮件与排查。
- 后续可增加“邮件模板表 + 批量投递表”，本期先不做。

## 4.2 容量规则
- 单账号邮箱容量上限 `100`（一期固定值）。
- 新邮件写入时：
  - 若已满，直接拒绝新邮件（返回容量已满）。

## 4.3 与背包联动
- 领取附件时执行：
  - 背包容量校验
  - 逐个道具入包
- 任一附件入包失败时，整封领取失败并回滚（事务一致性）。
- 领取失败需返回明确失败语义给客户端（用于提示）。

## 5. 服务端 API 设计（草案）

## 5.1 获取邮件列表
- `GET /api/game/mail/list`
- 返回：
  - `mails`（摘要列表，含状态、首附件）
  - `count`
  - `capacity`（100）
  - （建议）`unread_count`（便于红点）

## 5.2 获取邮件详情
- `GET /api/game/mail/detail?mail_id=...`
- 返回完整正文与附件。
- 在此接口内直接完成“标记已读”（若此前未读则置已读）。

## 5.3 领取邮件附件
- `POST /api/game/mail/claim`
- 入参：`mail_id`
- 语义：
  - 仅可对未领取且有附件邮件执行
  - 成功后 `is_claimed=true`
  - 返回本次领取物品汇总

## 5.4 删除邮件（统一批量删除接口）
- `POST /api/game/mail/delete`
- 入参建议：
  - `mail_ids: string[]`（可选）
  - `delete_mode: "manual" | "read_and_claimed"`（必填）
- 语义：
  - `manual`：删除指定 `mail_ids`，用于单封/多封手动删除
  - `read_and_claimed`：忽略 `mail_ids`，批量删除“已读且已领取”邮件（供“一键删除”按钮使用）
  - 删除实现采用软删：将 `is_deleted=true`（并记录 `deleted_at`），列表查询默认过滤软删邮件
- 删除约束：
  - 不允许删除“未读且未领取附件”的邮件
  - 无附件邮件按阅读态删除（已读可删，未读建议先读后删）

## 9. 已锁定规则（本版）

1. 删除接口统一为 `POST /api/game/mail/delete`，支持单封与批量两种模式。  
2. 列表页按钮文案固定为`一键删除`，点击后先弹确认。  
3. 附件最多10种不同物品；每种可堆叠（堆叠上限复用物品配置）；详情区每行4个。  
4. 邮件状态采用“双维度”：`已读/未读` + `无附件/未领取/已领取`。  
5. 领取与背包联动：服务端事务保证，任一失败整体回滚并返回失败原因。  
6. 邮箱满100时拒绝新邮件。  
7. 不允许删除“未读且未领取附件”的邮件。  
8. 点击详情即标记已读；无附件邮件隐藏领取按钮；已领取附件用灰化+角标样式。
