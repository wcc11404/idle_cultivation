# 邮箱系统

## 职责与边界

- 管理“设置 -> 邮箱”入口下的邮件列表、详情、领取、删除交互。
- 客户端不维护邮件真值，不在本地推导可领取状态，所有状态以服务端返回为准。
- 客户端仅负责会话内轮询刷新（30 秒）和界面渲染。
- 邮箱红点由 `GameUI` 统一聚合和渲染，`MailModule` 只上报摘要状态。

## 核心交互

1. 进入邮箱页后请求 `mail/list`，按时间倒序渲染缩略卡。
2. 点击缩略卡请求 `mail/detail`，服务端会同时标记已读。
3. 详情弹窗支持：
- 附件领取（无附件时隐藏领取按钮）
- 删除单封（走统一删除接口）
- 关闭弹窗
4. 一键删除按钮会先弹确认，再调用批量删除模式（删除“已读且非未领取”）。
5. 邮件详情弹窗内点击“删除”直接调用删除，不再二次确认。

## 列表与触控规则

- 邮件列表只允许纵向滚动，横向滚动禁用。
- 横纵滚动条隐藏。
- 移动端使用 touch-slop 策略：滑动优先，点击必须低位移阈值内触发，避免“点击吞滑动”。

## API

- `GET /game/mail/list`
- `GET /game/mail/detail?mail_id=...`
- `POST /game/mail/claim`
- `POST /game/mail/delete`

## 红点规则（一）

- 真值来源：`GET /game/mail/list` 返回的 `unread_count`
- 点亮条件：`unread_count > 0`
- 显示位置：
  - `设置 -> 邮箱` 按钮：大号数字红点，显示 `unread_count`
  - 主 `设置 Tab`：小红点透传，不显示数字
- 聚合边界：
  - `MailModule` 通过 `mail_state_changed(unread_count, total_count)` 上报最新摘要
  - `GameUI` 负责统一渲染入口按钮和 Tab 红点
- 数字显示约定：
  - `1-99` 显示实际数量
  - `>99` 显示 `99+`
- 即时刷新时机：
  - `refresh_mail_list()`
  - `refresh_indicator_only()`
  - 查看详情标记已读后
  - 领取附件后
  - 单封删除后
  - 一键删除后
- 轮询失败：
  - 保持上一次成功状态，不主动清空红点

## 会话内轮询（红点预埋）

- `GameUI` 内统一轮询周期 30 秒。
- 同一登录会话内串行调用：
1. `task_list`
2. `mail_list`
- 轮询失败静默，下个周期自动重试。
- 轮询结果仅更新轻量状态，不强制打断当前页面。

## 状态文案约定

- 已读标识：`[已读]` / `[未读]`
- 单封附件领取成功：`邮件附件领取成功`
- 单封删除成功：`邮件已删除`
- 单封删除拦截（有附件未领取）：`有未领取的附件，不能删除该邮件`
- 一键删除成功：`一键删除成功，已删除X封邮件`
- 一键删除空结果：`无可删除邮件（仅删除已读且非未领取状态的邮件）`
